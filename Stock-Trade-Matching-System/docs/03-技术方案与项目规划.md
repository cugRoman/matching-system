## 1. 技术选型与工具链

### 1.1 技术栈总览

| 层级 | 技术选型 | 说明 |
|------|---------|------|
| **后端核心** | Java 17+ / Spring Boot 3.x | 企业级框架，丰富的中间件生态 |
| **实时通信** | Spring WebSocket + STOMP | 行情推送、回报推送的实时通道 |
| **REST API** | Spring Web (MVC) | 订单提交、查询、管理等 CRUD 操作 |
| **前端框架** | Vue 3 + TypeScript | 组合式 API，类型安全 |
| **UI 组件库** | Element Plus | 中文友好，表格/表单组件完善 |
| **图表库** | ECharts | 行情K线、交易量可视化 |
| **实时缓存** | Redis 7.x | 订单簿内存态、对敲检测索引、行情快照 |
| **持久化存储** | PostgreSQL 16.x | 订单历史、成交记录、审计日志 |
| **消息队列** | 内置 Disruptor / 或 Redis Stream | 模块间异步消息传递，高吞吐低延迟 |
| **构建工具** | Maven / Gradle | 后端依赖管理与构建 |
| **前端构建** | Vite | 极速热更新，Vue 3 官方推荐 |
| **版本控制** | Git + GitHub/Gitee | 分支协作 |
| **接口文档** | Swagger / SpringDoc OpenAPI | API 自动生成文档 |
| **容器化** | Docker + Docker Compose | 一键启动全部服务（后端 + 前端 + Redis + PG） |

### 1.2 选型理由

#### 后端：Java / Spring Boot

- **开发效率**：Spring Boot 自动配置 + 注解驱动大幅减少样板代码，6 人团队 2~3 周可完成
- **并发模型**：`java.util.concurrent` 提供成熟的并发工具（`ConcurrentHashMap`、`Lock`、`CompletableFuture`）
- **生态优势**：Spring Data JPA（PostgreSQL）、Spring Data Redis、Spring WebSocket 开箱即用
- **性能足够**：JIT 编译后的 Java 在非极致低延迟场景下性能完全满足需求；Disruptor 等无锁队列框架可达百万级吞吐
- **团队因素**：Java 是高校教学最广泛的语言之一，团队上手成本最低

#### 前端：Vue 3 + Element Plus

- **学习曲线平缓**：Vue 的模板语法直观，配合 Element Plus 可快速搭建管理界面
- **中文生态**：Element Plus 中文文档完善，社区活跃
- **适合数据密集型**：Element Plus 的 `el-table` 支持大数据量虚拟滚动，适合订单列表展示

#### 存储：Redis + PostgreSQL

- **Redis**：承载实时运算的全部热数据——订单簿（OrderBook）、对敲检测索引、行情快照、订单状态机
- **PostgreSQL**：持久化全部历史数据——订单流水、成交记录、撤单记录、审计日志
- **分工清晰**：写入路径为 `Redis（实时处理）→ 异步落盘 → PostgreSQL（离线分析）`

## 2. 系统总体架构

### 2.1 系统形态定义

本系统是一个**完整的 Web 应用**，包含：

- **交易客户端**（Web 前端）：用户通过浏览器下单、查看回报、查看行情
- **企业监控端**（Web 前端）：运营人员监控系统状态、查看订单流水、手动干预
- **交易后端服务**（Java 服务）：接收订单、校验、风控、撮合、回报推送
- **模拟交易所**（内嵌模块/独立服务）：模拟交易所行为，回复确认/成交/撤单回报
- **行情模拟器**（独立模块）：生成/播放模拟行情数据

### 2.3 数据流全景

#### 正向数据流（订单处理）

```
用户浏览器
  │ REST POST /api/orders  或  WebSocket SEND /app/order
  ▼
API Gateway (Controller)
  │ 反序列化 JSON → OrderDTO
  ▼
OrderValidator
  │ 字段校验 → 通过/拒绝
  ▼
RiskController (对敲检测)
  │ 查 Redis: hash(shareHolderId + market + securityId)
  │ 有反向挂单 && 价格可成交？
  ├─ 否 → ExchangeProxy.forward(order)
  │        │ 存入 Redis 订单簿
  │        │ 转发到模拟交易所
  │        ▼
  │     ExchangeSimulator → 回复确认/成交回报
  │
  └─ 是 → MatchingEngine.match(incomingOrder, counterOrders)
           │ 生成成交回报 (买方+卖方)
           │ 通过 ExchangeProxy 撤回对手方已发交易所的订单
           ▼
        ReportDispatcher
           │ 推送成交回报给客户 (WebSocket)
           │ 异步写入 PostgreSQL
           ▼
        用户浏览器收到实时回报
```

#### 反向数据流（回报处理）

```
模拟交易所 / 内部撮合引擎
  │ 回报消息 (确认/成交/撤单确认/拒绝)
  ▼
ReportDispatcher
  │ 更新 Redis 中的订单状态
  │ 异步持久化到 PostgreSQL
  │ 通过 WebSocket 推送给对应客户
  ▼
用户浏览器实时更新
```

## 3. 模块化功能设计

### 3.1 模块总览

系统分为 **7 个核心后端模块** + **2 个前端应用** + **2 个基础设施服务**：

| 编号 | 模块名 | 职责 | 关键类/接口 |
|------|--------|------|-----------|
| M1 | `gateway` | HTTP/WebSocket 接入，请求解析与响应 | `OrderController`, `WebSocketHandler` |
| M2 | `validator` | 订单字段合法性校验 | `OrderValidator`, `ValidationRule` |
| M3 | `risk-control` | 对敲交易检测与拦截 | `WashTradeDetector`, `OrderIndex` |
| M4 | `matching-engine` | 内部撮合引擎 + 成交价算法 | `MatchingEngine`, `PricingStrategy` |
| M5 | `exchange-proxy` | 与模拟交易所通信，管理已发送订单 | `ExchangeProxy`, `SentOrderRegistry` |
| M6 | `report-dispatcher` | 回报分发、状态更新、WebSocket 推送 | `ReportDispatcher`, `OrderStateManager` |
| M7 | `exchange-simulator` | 模拟交易所行为（确认、撮合、撤单） | `ExchangeSimulator`, `SimOrderBook` |
| F1 | `web-trader` | 交易客户端前端 | Vue 3 SPA |
| F2 | `web-admin` | 企业监控端前端 | Vue 3 SPA |
| I1 | `market-data` | 行情模拟与分发 | `MarketDataSimulator`, `MarketDataStore` |
| I2 | `persistence` | 异步持久化层与数据分析查询 | `TradeRepository`, `AnalyticsService` |

### 3.2 各模块详细设计

#### M1: Gateway（网关接入层）

**职责**：
- 提供 REST API 端点（下单、撤单、查询）
- 提供 WebSocket 端点（实时回报推送、行情推送）
- 请求反序列化（JSON → DTO）与响应序列化
- 基础限流与认证（可选）

**核心接口**：
```
POST   /api/orders          → 提交交易订单
DELETE /api/orders/{id}      → 提交撤单请求
GET    /api/orders           → 查询订单列表
GET    /api/orders/{id}      → 查询订单详情
GET    /api/executions       → 查询成交记录

WebSocket /ws/trade          → 实时回报通道
WebSocket /ws/market-data    → 实时行情通道
```

**关键设计**：
- REST 用于请求-响应式操作（下单、查询）
- WebSocket (STOMP) 用于服务端主动推送（回报、行情更新、告警）
- 使用 `@Validated` 注解实现请求体的初步格式校验

#### M2: Validator（订单校验模块）

**职责**：
- 实现责任链模式的校验流水线
- 每条校验规则独立封装，易于扩展

**校验规则链**：

| 顺序 | 校验规则 | rejectCode | 说明 |
|------|---------|-----------|------|
| 1 | `RequiredFieldRule` | 1001 | `clOrderId`, `shareHolderId` 非空 |
| 2 | `MarketRule` | 1002 | `market` ∈ {XSHG, XSHE, BJSE} |
| 3 | `SecurityIdRule` | 1003 | `securityId` 为 6 位数字字符 |
| 4 | `SideRule` | 1004 | `side` ∈ {B, S} |
| 5 | `QuantityRule` | 1005 | `qty > 0`；买入时 `qty % 100 == 0` |
| 6 | `PriceRule` | 1006 | `price > 0`；精度不超过 0.01 |
| 7 | `DuplicateOrderRule` | 1007 | `clOrderId` 不可重复 |
| 8 | `PriceLimitRule`（高级） | 1008 | 价格在 [跌停价, 涨停价] 内 |

**关键设计**：
- 使用 `interface ValidationRule` + `List<ValidationRule>` 责任链
- 校验失败立即短路返回，附带 `rejectCode` + `rejectText`
- 每条规则可独立单元测试

#### M3: RiskControl（对敲风控模块）

**职责**：
- 维护**按 (shareHolderId, market, securityId) 索引的待成交订单表**
- 新订单到达时，查找是否存在同一股东号 + 同一市场 + 同一股票的反向挂单
- 判断价格是否可成交（买价 ≥ 卖价）
- 命中则触发拦截 + 撮合流程

**核心数据结构（Redis）**：

```
Key:   wash:{shareHolderId}:{market}:{securityId}
Value: Hash
       field = side (B/S)
       每个 side 下存放该方向的活跃订单列表（按价格排序）
```

或使用 Java 内存结构（推荐，性能更优）：

```
ConcurrentHashMap<WashKey, OrderBook>
  WashKey = (shareHolderId, market, securityId)
  OrderBook = {buyOrders: TreeMap<Price, Queue<Order>>,
               sellOrders: TreeMap<Price, Queue<Order>>}
```

**并发安全**：
- 按 `WashKey` 粒度加锁（`ReentrantLock` 或 `synchronized`），保证同一股东号+同一股票的订单串行处理
- 不同股东号/不同股票的订单之间无锁竞争，可完全并行

**关键设计**：
- 新订单处理流程：`查找反向挂单 → 有且可成交 → 拦截 + 调用 MatchingEngine → 无 → 注册到索引 + 转发`
- 订单成交/撤销后需从索引中移除

#### M4: MatchingEngine（内部撮合引擎）

**职责**：
- 接收对敲命中的订单对，执行配对撮合
- 实现成交价生成算法
- 处理部分成交、零股、多対手方连续配对
- 生成成交回报（买方 + 卖方各一份，共享同一 `execId`）

**成交价算法**（分层策略）：

| 阶段 | 策略 | 逻辑 |
|------|------|------|
| 基础目标 | 先到方价格（被动方报价） | `execPrice = 先到订单的 price` |
| 高级目标 | 行情约束价 | `execPrice = clamp(先到方价格, bidPrice, askPrice)` |

**撮合流程**：

```
1. 从 RiskControl 获取全部可配对的反向挂单（已按价格优先排序）
2. 按 价格优先 → 时间优先 逐个配对
3. 每次配对：
   a. 成交数量 = min(incoming.remainQty, counter.remainQty)
   b. 成交价 = 按定价策略计算
   c. 生成 ExecReport (买方) + ExecReport (卖方)，共享 execId
   d. 更新双方 remainQty
   e. 若 counter 全部成交，从索引移除；若 incoming 全部成交，结束
4. incoming 若仍有剩余，注册到订单簿等待后续配对
```

**`execId` 生成**：
- 格式：`EXEC_{12位递增序号}`，如 `EXEC_000000001`
- 使用 `AtomicLong` 保证唯一性

#### M5: ExchangeProxy（交易所代理模块）

**职责**：
- 维护已发送到交易所的订单注册表
- 转发通过风控的订单到模拟交易所
- 发送撤单请求（对敲命中后撤回对手方订单）
- 接收交易所回报并分发

**关键设计**：
- 内部通信方式：与 `ExchangeSimulator` 之间使用内存消息队列（Disruptor / `BlockingQueue`）
- 维护 `SentOrderRegistry`：记录每笔已发送订单的状态，用于撤单时快速查找
- 撤单流程：`生成撤单请求 → 发送到交易所 → 等待撤单回报 → 更新状态`

#### M6: ReportDispatcher（回报分发模块）

**职责**：
- 统一回报出口——无论来自交易所回报还是内部撮合生成的回报
- 更新 Redis 中的订单状态（状态机跃迁）
- 通过 WebSocket 向对应客户推送实时回报
- 异步将回报写入 PostgreSQL 持久化

**订单状态机管理**：
- 使用枚举 `OrderStatus` + 状态转换表驱动
- 每次状态变更记录日志（含时间戳、触发事件、前后状态）

**WebSocket 推送**：
- 使用 STOMP 协议的 Topic 订阅模型
- 客户端订阅 `/topic/reports/{shareHolderId}` 接收自己的回报
- 监控端订阅 `/topic/admin/reports` 接收全部回报流

#### M7: ExchangeSimulator（模拟交易所）

**职责**：
- 模拟交易所的基本行为：接收订单 → 确认 → 排队/撮合 → 回报
- 提供可配置的行为策略（简单确认 / 简单撮合 / 延迟模拟）

**模拟策略**：

| 级别 | 行为 | 所需实现 |
|------|------|---------|
| Level 1（基础） | 收到订单即回复确认；撤单即确认 | 简单回调 |
| Level 2（进阶） | 维护简单订单簿，不同股东号间可撮合 | 基础 OrderBook |
| Level 3（完整） | 完整的价格优先/时间优先撮合引擎 | 完整 OrderBook |

**推荐**：基础目标阶段实现 Level 1，有余力则升级到 Level 2。

**页面结构**：

| 仪表盘 | 系统概览（订单量、成交量、对敲拦截数、延迟指标） |
| 订单监控 | 全部订单实时流水，支持按股东号/股票筛选 |
| 对敲告警 | 被拦截的对敲订单列表及详情 |
| 性能监控 | 吞吐量/延迟的实时图表（P50/P95/P99） |
| 手动下单 | 管理员手动提交订单入口（用于测试） |
| 行情管理 | 行情数据的导入/编辑/查看 |
| 压力测试控制台 | 配置并发数、订单量、对敲比例，启动压力测试 |

#### I1: MarketData（行情模拟模块）

**职责**：
- 生成/加载模拟行情数据（bidPrice, askPrice）
- 维护行情快照存储（Redis）
- 通过 WebSocket 向前端推送行情更新

**行情生成策略**：
- **静态模式**：从 JSON 文件加载预设行情
- **动态模式**：基于随机游走模型模拟行情变动（开盘价 ± 随机波动，受涨跌停限制）

#### I2: Persistence（持久化与分析模块）

**职责**：
- 异步将订单、成交记录落盘到 PostgreSQL
- 提供历史数据查询 API
- 提供数据分析功能（对敲命中率、成交量分布、延迟统计等）

**关键设计**：
- 使用异步写入（`@Async` 或独立线程池），不阻塞核心交易路径
- 批量插入优化（每 100 条或每 100ms 刷一次盘）

### 4.2 目录结构设计理由

| 设计决策 | 理由 |
|---------|------|
| 后端单体项目，内部按模块分包 | 2~3 周时间不适合做微服务拆分，单体内模块化即可；通过接口隔离保证未来可拆 |
| 前端交易端+监控端合并为一个 SPA | 共享组件和 API 封装，通过路由区分；两端分别实现页面 |
| 测试数据独立目录 | 测试数据与代码分离，便于批量生成和管理 |
| Docker Compose 一键启动 | 降低环境搭建成本，评审和演示时一条命令即可启动全栈 |

### 5.2 Redis 数据结构设计

| Key Pattern | 数据结构 | 用途 | TTL |
|-------------|---------|------|-----|
| `order:{clOrderId}` | Hash | 订单完整信息 + 状态 | 交易日结束后清理 |
| `book:{shareHolderId}:{market}:{securityId}:{side}` | Sorted Set | 按价格排序的活跃订单（对敲索引） | 无（主动清理） |
| `mkt:{market}:{securityId}` | Hash | 行情快照 (bidPrice, askPrice) | 覆盖更新 |
| `stats:throughput` | String (计数器) | 吞吐量统计 | 每分钟重置 |
| `stats:latency` | List | 延迟采样值 | 滚动窗口 |

> **注意**：核心对敲检测逻辑推荐使用 Java 内存数据结构（`ConcurrentHashMap` + `TreeMap`）以获得最低延迟，Redis 作为持久化备份和跨实例共享的补充。单实例部署时优先使用内存结构。

## 8. 测试方案

### 8.1 测试分层

```
                    ┌────────────────────────┐
                    │    端到端测试 (E2E)      │   ← 少量：验证完整用户流程
                    │   前端 → API → 后端      │
                    ├────────────────────────┤
                    │    集成测试              │   ← 中等：验证模块间协作
                    │   多模块联合测试          │
                    ├────────────────────────┤
                    │    单元测试              │   ← 大量：验证每个模块/规则
                    │   Validator/Risk/Match  │
                    └────────────────────────┘
```

### 8.2 单元测试要求

| 模块 | 关键测试内容 | 最低覆盖要求 |
|------|------------|------------|
| Validator | 每条校验规则的正反例 | 每条规则 ≥ 3 个用例 |
| RiskControl | 对敲检测的 9 个边界场景（见文档二 §1.1） | 全部 9 个场景 |
| MatchingEngine | 等量成交、部分成交、零股、多对手、成交价 | ≥ 10 个用例 |
| OrderStateManager | 状态转换的合法/非法路径 | 所有合法转换 + 主要非法转换 |

### 8.3 集成测试场景

| 编号 | 场景 | 验证要点 |
|------|------|---------|
| IT-01 | 正常订单完整生命周期 | 下单→确认→成交→回报推送→数据库记录 |
| IT-02 | 校验失败订单 | 下单→拒绝回报→未转发到交易所 |
| IT-03 | 对敲订单完整流程 | 买单→转发→卖单→拦截→内部撮合→成交回报→撤单 |
| IT-04 | 部分成交+剩余处理 | 大买单 vs 小卖单→部分成交→剩余留存 |
| IT-05 | 撤单流程 | 下单→请求撤单→撤单确认/拒绝 |
| IT-06 | 行情约束撮合 | 导入行情→对敲订单→成交价在行情区间内 |

### 8.4 性能测试方案

#### 吞吐量测试

- **工具**：后端内置压力测试控制台（`/api/admin/stress-test`）或使用 JMH
- **方法**：生成 N 笔订单（可配置对敲比例），计时处理，计算 TPS
- **指标**：
  - 基础目标：≥ 1,000 TPS
  - 高级目标：≥ 100,000 TPS
- **展示**：前端性能仪表盘实时显示 TPS 曲线

#### 延迟测试

- **方法**：在订单接收入口和回报发送出口之间打高精度时间戳
- **指标**：统计 P50 / P95 / P99 / Max 延迟
- **展示**：前端延迟分布直方图

#### 压力测试数据集配置

```json
{
  "totalOrders": 100000,
  "concurrency": 50,
  "stockCount": 100,
  "shareholderCount": 200,
  "washTradeRatio": 0.2,
  "cancelRatio": 0.05,
  "priceRange": { "min": 5.0, "max": 100.0 },
  "qtyRange": { "min": 100, "max": 10000, "step": 100 }
}
```

### 8.5 测试数据生成器

使用 Python 脚本生成标准化测试数据（`test-data/generator/generate_orders.py`），支持：

- 生成指定数量的随机合法订单
- 按比例插入对敲订单对
- 按比例插入非法订单（用于校验测试）
- 生成配套的行情数据
- 输出为 JSON 文件，可通过 API 批量导入

### 9.2 演示流程建议

#### 演示一：基础功能展示（~10 分钟）

1. **正常下单**：在交易客户端输入一笔合法买单，展示订单确认回报
2. **校验拦截**：输入一笔非法订单（如价格为负），展示拒绝回报和错误码
3. **对敲拦截+内部撮合**：
   - 用同一股东号提交一笔买单（成功转发）
   - 再提交一笔同一股票的卖单 → 展示对敲拦截 → 展示内部成交回报
   - 切换到监控端展示对敲告警记录
4. **回报实时推送**：展示 WebSocket 实时推送到多个浏览器窗口

#### 演示二：高级功能展示（~5 分钟）

5. **行情约束**：导入行情数据，展示成交价被约束在行情区间内
6. **撤单**：提交订单后发起撤单，展示撤单流程
7. **数据分析**：展示历史订单统计、对敲命中率图表

#### 演示三：性能展示（~5 分钟）

8. **压力测试**：在监控端启动压力测试，实时观察：
   - TPS 曲线上升到峰值
   - 延迟分布（P50/P95/P99）
   - 订单流水的实时刷新
9. **性能对比**：展示优化前 vs 优化后的性能指标

## 附录 A：核心设计决策记录

| 决策项 | 选定方案 | 备选方案 | 决策理由 |
|--------|---------|---------|---------|
| 后端语言 | Java / Spring Boot | C++, Go, Rust | 团队熟悉度最高，生态最完善，2~3周可交付 |
| 前端框架 | Vue 3 + Element Plus | React + Ant Design | 学习曲线低，中文生态友好 |
| 实时通信 | WebSocket + STOMP | gRPC-Web, 轮询 | 实时性好且浏览器原生支持 |
| 缓存 | Redis + Java 内存 | 纯 Redis | 核心路径用内存确保最低延迟，Redis 做持久化备份 |
| 数据库 | PostgreSQL | MySQL, SQLite | JSONB 支持好，适合审计日志；ACID 完整 |
| 成交定价 | 先到方价格（基础） / 行情约束（高级） | 中间价 | 符合交易所惯例，避免精度问题 |
| 对敲检测并发 | 按 WashKey 粒度加锁 | 全局锁, 单线程 | 兼顾正确性与并行度 |
| 模块间通信 | Pipeline + 事件回调 | 消息队列, 直接调用 | 主流程清晰，分支灵活 |
| 交易所模拟 | Level 1 简单确认（基础） | 完整撮合引擎 | 时间有限，核心重心在风控撮合 |
