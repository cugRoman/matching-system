## 1. A 股交易基础知识

### 1.1 A 股主板市场概述

A 股（人民币普通股票）是在中国境内上市、以人民币标价和交易的股票。本项目涉及的 A 股主板市场涵盖三家交易所：

| 交易所 | 代码 | 全称 | 股票代码特征 | 说明 |
|--------|------|------|-------------|------|
| 上交所 | `XSHG` | 上海证券交易所 | 6 开头（如 600030） | 主板，大盘蓝筹股为主 |
| 深交所 | `XSHE` | 深圳证券交易所 | 0 开头（如 000001） | 主板，中小型公司为主 |
| 北交所 | `BJSE` | 北京证券交易所 | 8 开头（如 830799） | 面向创新型中小企业 |

> **说明**：本项目模拟的是 A 股主板交易，不涉及科创板、创业板等特殊板块。

### 1.2 竞价撮合机制

A 股市场采用**连续竞价交易**（集合竞价时段暂不考虑），核心原则是：

#### 价格优先
- **买方**：出价**高**的订单优先成交
- **卖方**：出价**低**的订单优先成交
- 直觉理解：愿意付更多钱的买家、愿意接受更低价格的卖家，会被优先撮合

#### 时间优先
- 在价格相同时，**先到达**交易所的订单优先成交
- 同价格的订单按照到达时间排列形成队列

#### 成交条件
- 当**最高买价 ≥ 最低卖价**时，撮合发生
- 成交价的确定：
  - 限价单 vs 限价单：通常取挂单方（先到的订单）的价格
  - 核心原则是保护先挂单的一方获得其报价

#### 举例说明

假设当前某只股票订单簿如下：

```
卖方挂单（从低到高）：
  卖1: 10.50 元 × 200 股  (最先挂单)
  卖2: 10.60 元 × 500 股

买方挂单（从高到低）：
  买1: 10.40 元 × 300 股  (最先挂单)
  买2: 10.30 元 × 100 股
```

此时最高买价 10.40 < 最低卖价 10.50，不满足成交条件，无撮合发生。

现在新来一笔**买入 10.55 元 × 300 股**的订单：
- 最高买价变为 10.55 ≥ 最低卖价 10.50，触发撮合
- 与卖1（10.50 × 200 股）撮合：成交价 10.50 元，成交 200 股
- 该买单剩余 100 股，10.55 ≥ 10.60 不成立，不再撮合，剩余 100 股挂在买方队列

### 1.3 交易单位

| 概念 | 说明 |
|------|------|
| **手** | A 股最小交易单位，1 手 = 100 股 |
| **整手** | 100 股的整数倍（100, 200, 300...） |
| **零股** | 不足 100 股的部分（1~99 股） |

**A 股交易单位规则**：
- **买入**：必须以 **整手**（100 股的整数倍）为单位下单
- **卖出**：可以卖出零股，但零股必须**一次性全部卖出**
  - 例如：持有 150 股，可以卖 100 股（剩余 50 股），也可以卖 150 股
  - 不能拆分为两次零股卖出（如先卖 30 股再卖 20 股）——但若持有的零股就是通过之前整手卖出剩余的，则可以一次卖出全部零股
- **零股的来源**：通常由部分成交产生，或高送转、配股等产生

### 1.4 价格相关规则

| 规则 | 主板（沪深） | 北交所 |
|------|-------------|--------|
| 涨跌幅限制 | ±10%（ST 股票 ±5%） | ±30% |
| 最小变动价位 | 0.01 元 | 0.01 元 |

- **涨停价** = 前收盘价 × (1 + 涨跌幅限制)，按 0.01 元取整（四舍五入）
- **跌停价** = 前收盘价 × (1 - 涨跌幅限制)，按 0.01 元取整
- 订单价格必须在 [跌停价, 涨停价] 区间内

> **本项目简化**：由于我们是委前风控+内部撮合系统，涨跌停校验依赖行情数据。在高级目标"行情接入"实现前，基础目标阶段可暂不做涨跌停校验，仅做基本字段合法性校验。

## 2. 对敲交易的定义与风险

### 2.1 什么是对敲交易

**对敲交易**（Wash Trading），是指**同一控制人**（在本项目中体现为同一股东号 `shareHolderId`）对**同一只股票**同时或交替下达**买入和卖出订单**，使该股票在市场上产生成交记录，但实质上资产并未真正转手。

#### 对敲交易的核心判定要素

```
对敲交易 = 同一股东号
         + 同一市场（market）
         + 同一股票代码（securityId）
         + 存在反向买卖方向（一笔 Buy，一笔 Sell）
         + 价格可成交（买价 ≥ 卖价）
```

#### 举例

股东号为 `SH00001234` 的客户在短时间内提交了两笔订单：

```json
// 订单 A：买入
{
  "clOrderId": "ORD_000001",
  "market": "XSHG",
  "securityId": "600030",
  "side": "B",
  "qty": 500,
  "price": 25.50,
  "shareHolderId": "SH00001234"
}

// 订单 B：卖出
{
  "clOrderId": "ORD_000002",
  "market": "XSHG",
  "securityId": "600030",
  "side": "S",
  "qty": 300,
  "price": 25.40,
  "shareHolderId": "SH00001234"
}
```

分析：
- 同一股东号 `SH00001234` ✓
- 同一市场 `XSHG` ✓
- 同一股票 `600030` ✓
- 反向买卖（B vs S）✓
- 买价 25.50 ≥ 卖价 25.40，可成交 ✓

→ **判定为对敲交易**，后到的订单应被拦截。

### 2.2 非对敲的场景辨析

| 场景 | 是否对敲 | 理由 |
|------|---------|------|
| 同股东号 + 同股票 + 同方向（都是买） | ❌ | 无反向配对 |
| 同股东号 + 不同股票 + 反向 | ❌ | 不同标的 |
| 不同股东号 + 同股票 + 反向 | ❌ | 不同控制人 |
| 同股东号 + 同股票 + 反向 + 买价 < 卖价 | ❌ | 价格不可能成交* |
| 同股东号 + 同市场 + 同股票 + 反向 + 买价 ≥ 卖价 | ✅ | 满足所有条件 |

> *关于"买价 < 卖价是否仍算对敲"的设计选择：严格来说，价格不可成交的反向订单不会在交易所触发撮合，因此不构成实际对敲。但在保守风控策略下，也可选择无论价格是否可成交都进行拦截。**本项目建议采用严格判定（买价 ≥ 卖价才算对敲）**，这更贴近实际业务。

### 2.3 为什么要禁止对敲交易

1. **制造虚假成交量**：让市场误以为该股票交投活跃，误导投资者决策
2. **操纵股价**：通过自买自卖控制成交价格，达到拉升或打压股价的目的
3. **洗钱风险**：通过交易手段转移资金
4. **违反《证券法》**：对敲交易属于操纵市场行为，是法律明确禁止的

### 2.4 券商侧的风控责任

- 交易所对提交对敲交易的券商会进行处罚
- 因此券商需要在**订单送达交易所之前**（即"委前"）进行风控检查
- 本系统就是扮演这个"委前风控网关"的角色

## 3. 系统在交易链路中的定位

### 3.1 完整交易链路

在实际证券交易中，一笔订单的完整路径如下：

```
  ┌──────────┐     ┌──────────────────────────┐     ┌──────────┐
  │  交易客户  │────▶│     本系统（委前风控+撮合）    │────▶│   交易所   │
  │ (下单方)  │◀────│                          │◀────│  (撮合主机) │
  └──────────┘     └──────────────────────────┘     └──────────┘
     发出订单        接收 → 校验 → 风控 → 决策         接收合法订单
     接收回报        转发订单 / 拦截 + 内部撮合          返回回报
```

### 3.2 本系统的双重角色

#### 角色一：风控网关（Gateway + Risk Control）

作为交易客户与交易所之间的"关卡"：

- **正向（订单方向）**：接收客户订单 → 校验合法性 → 检测对敲 → 合法则转发到交易所
- **反向（回报方向）**：接收交易所回报 → 转发给客户

#### 角色二：内部撮合引擎（Matching Engine）

当检测到对敲时，不能简单地丢弃订单，而是需要：

1. **拦截**后到的订单，不发往交易所
2. 在**内部完成撮合**，生成模拟成交回报
3. 如果先到的对手方订单已经发送到交易所，需要**向交易所发送撤单请求**，将其撤回
4. 将成交回报和撤单回报返回给交易客户

> **为什么不能只拦截不撮合？**  
> 因为客户确实有交易需求（虽然是同一股东号，可能是程序化交易的不同策略同时产生了买卖信号）。拦截后在内部完成撮合，既避免了对敲违规，又满足了客户的交易需求。

| 目标 | 消息类型 | 说明 |

| 交易客户 | 系统生成的成交回报 | 内部撮合产生的成交结果 |

## 4. 订单全生命周期流转

### 4.1 场景一：正常订单（无对敲）

交易客户提交一笔合法订单，同一股东号下不存在反向订单。

```
时间线 ──────────────────────────────────────────────────────────▶

交易客户                 本系统                           交易所
   │                      │                               │
   │  ① 提交买入订单       │                               │
   │ ─────────────────▶   │                               │
   │                      │  ② 校验通过                    │
   │                      │  ③ 对敲检测：未命中             │
   │                      │  ④ 转发订单                    │
   │                      │ ─────────────────────────────▶│
   │                      │                               │  ⑤ 交易所确认
   │                      │  ⑥ 收到确认回报                │
   │                      │ ◀─────────────────────────────│
   │  ⑦ 收到确认回报       │                               │
   │ ◀─────────────────── │                               │
   │                      │                               │  ... (等待撮合)
   │                      │  ⑧ 收到成交回报                │
   │                      │ ◀─────────────────────────────│
   │  ⑨ 收到成交回报       │                               │
   │ ◀─────────────────── │                               │
```

**关键要点**：
- 本系统只做"透传"：订单透传给交易所，回报透传给客户
- 系统需要记录已转发的订单信息，用于后续可能的对敲检测

### 4.2 场景二：订单校验失败

```
交易客户                 本系统
   │                      │
   │  ① 提交订单（字段非法）│
   │ ─────────────────▶   │
   │                      │  ② 校验失败（如价格为负）
   │  ③ 收到非法回报       │
   │ ◀─────────────────── │  ← 系统自行生成非法回报
   │                      │
   │              （订单未转发至交易所）
```

**校验内容包括但不限于**：
- `clOrderId` 不为空
- `market` 必须是 `XSHG`、`XSHE`、`BJSE` 之一
- `securityId` 必须是 6 位数字
- `side` 必须是 `B` 或 `S`
- `qty` 必须大于 0
- `price` 必须大于 0
- `shareHolderId` 不为空
- 数量精度校验（买入是否为 100 的整数倍等）

### 4.3 场景三：检测到对敲交易

这是本系统最核心的场景。假设同一股东号 `SH00001234` 先后提交了两笔订单：

```
时间线 ──────────────────────────────────────────────────────────▶

交易客户                 本系统                           交易所
   │                      │                               │
   │  ① 提交买入订单 A     │                               │
   │ ─────────────────▶   │                               │
   │                      │  ② 校验通过                    │
   │                      │  ③ 对敲检测：无反向订单，通过    │
   │                      │  ④ 转发订单 A                  │
   │                      │ ─────────────────────────────▶│
   │                      │     （记录订单 A 的信息）        │
   │                      │                               │
   │  ⑤ 提交卖出订单 B     │                               │
   │    (同股东号+同股票)   │                               │
   │ ─────────────────▶   │                               │
   │                      │  ⑥ 校验通过                    │
   │                      │  ⑦ 对敲检测：命中！             │
   │                      │     订单 A (买) vs 订单 B (卖)  │
   │                      │     同股东号 + 同股票 + 反向     │
   │                      │                               │
   │                      │  ⑧ 拦截订单 B（不转发到交易所）  │
   │  ⑨ 收到非法回报       │  ← 生成对敲拦截的非法回报       │
   │ ◀─────────────────── │                               │
   │                      │                               │
   │                      │  ⑩ 内部撮合：A 与 B 配对        │
   │                      │     计算成交价、成交量           │
   │  ⑪ 收到成交回报       │  ← 生成买方成交回报             │
   │ ◀─────────────────── │                               │
   │  ⑫ 收到成交回报       │  ← 生成卖方成交回报             │
   │ ◀─────────────────── │                               │
   │                      │                               │
   │                      │  ⑬ 向交易所发送撤单请求          │
   │                      │     （撤回已发送的订单 A）        │
   │                      │ ─────────────────────────────▶│
   │                      │                               │  ⑭ 交易所处理撤单
   │                      │  ⑮ 收到撤单确认回报             │
   │                      │ ◀─────────────────────────────│
   │  ⑯ 收到撤单确认回报   │                               │
   │ ◀─────────────────── │                               │
```

**关键要点**：
1. 先到的订单 A 已经被转发到交易所 → 需要撤回
2. 后到的订单 B 被拦截 → 不转发，系统生成非法回报
3. 系统内部完成 A 和 B 的撮合 → 生成成交回报（买方 + 卖方各一份）
4. 撤单和成交回报的时序：应先完成内部撮合并生成回报，再异步发送撤单
5. **复杂情况**：如果订单 A 在交易所已经部分成交了怎么办？需要考虑已成交部分不可撤、仅撤回剩余部分的情况

### 4.4 场景四：部分成交撮合

当两笔对敲订单的数量不同时：

```
订单 A：买入 600030，500 股，25.50 元（股东号 SH00001234）
订单 B：卖出 600030，300 股，25.40 元（股东号 SH00001234）
```

撮合结果：
- 成交数量：300 股（取较小值）
- 订单 A 剩余：200 股（可能继续挂在系统中等待后续匹配，或继续转发到交易所）

### 4.5 场景五：撤单处理（高级目标）

```
交易客户                 本系统                           交易所
   │                      │                               │
   │  ① 提交撤单请求       │                               │
   │ ─────────────────▶   │                               │
   │                      │  ② 检查：原始订单是否存在？      │
   │                      │  ③ 检查：订单当前状态？          │
   │                      │     - 若已全部成交/已撤销 → 拒绝  │
   │                      │     - 若在内部等待 → 内部撤回     │
   │                      │     - 若已发送到交易所 → 转发撤单 │
   │                      │ ─────────────────────────────▶│
   │                      │                               │
```

## 5. 数据结构与消息详解

### 5.1 消息分类总览

本系统涉及的全部消息类型可分为三组：

```
┌─ 客户 → 系统（输入）──────────────────────────────┐
│  ① 交易订单 (Order)                               │
│  ② 撤单请求 (Cancel Request)                      │
├─ 交易所 → 系统（输入）────────────────────────────┤
│  ③ 确认单据回报 (Confirm)                          │
│  ④ 订单非法回报 (Order Reject)                     │
│  ⑤ 成交回报 (Execution Report)                    │
│  ⑥ 撤单确认回报 (Cancel Confirm)                   │
│  ⑦ 撤单非法回报 (Cancel Reject)                    │
├─ 行情源 → 系统（输入）────────────────────────────┤
│  ⑧ 行情信息 (Market Data)                         │
└───────────────────────────────────────────────────┘
```

> **注意**：本系统的输出消息复用上述结构——合法订单转发给交易所、各类回报转发给客户、系统自生成的回报也使用同样的结构。

### 5.2 交易订单（Order）

客户发送给本系统的买入/卖出请求。

| 字段 | 类型 | 说明 | 校验规则 |
|------|------|------|---------|
| `clOrderId` | char[16] | 订单唯一编号 | 非空，唯一 |
| `market` | char[4] | 市场 | `XSHG` / `XSHE` / `BJSE` |
| `securityId` | char[6] | 股票代码 | 6 位数字字符 |
| `side` | char[1] | 买卖方向 | `B`(Buy) / `S`(Sell) |
| `qty` | uint32 | 数量（股） | > 0，买入需为 100 的整数倍 |
| `price` | double | 价格（元） | > 0，精度 0.01 |
| `shareHolderId` | char[10] | 股东号 | 非空 |

**JSON 示例**：
```json
{
  "clOrderId": "ORD_000001",
  "market": "XSHG",
  "securityId": "600030",
  "side": "B",
  "qty": 500,
  "price": 25.50,
  "shareHolderId": "SH00001234"
}
```

> ⚠️ **任务书勘误**：任务书 3.1 节中 `B` 标注为"卖"、`S` 标注为"买"，与行业惯例和任务书 3.6/3.7 节的标注矛盾。本项目统一约定：**`B` = Buy = 买入，`S` = Sell = 卖出**。

### 5.3 撤单请求（Cancel Request）

客户请求撤回已提交的订单（高级目标）。

| 字段 | 类型 | 说明 |
|------|------|------|
| `orderId` | char[16] | 撤单请求的唯一编号 |
| `origClientId` | char[16] | 待撤原始订单的编号 |
| `market` | char[4] | 待撤订单的市场 |
| `securityId` | char[6] | 待撤订单的股票代码 |
| `shareHolderId` | char[10] | 待撤订单的股东号 |
| `side` | char[1] | 待撤订单的买卖方向 |

**JSON 示例**：
```json
{
  "orderId": "CXL_000001",
  "origClientId": "ORD_000001",
  "market": "XSHG",
  "securityId": "600030",
  "shareHolderId": "SH00001234",
  "side": "B"
}
```

### 5.4 行情信息（Market Data）

实时行情快照，以 JSON 数组形式批量输入（高级目标）。

| 字段 | 类型 | 说明 |
|------|------|------|
| `market` | char[4] | 市场 |
| `securityId` | char[6] | 股票代码 |
| `bidPrice` | double | 当前最优买价（买方愿出的最高价） |
| `askPrice` | double | 当前最优卖价（卖方愿接受的最低价） |

**JSON 示例**：
```json
[
  {
    "market": "XSHG",
    "securityId": "600030",
    "bidPrice": 25.45,
    "askPrice": 25.50
  },
  {
    "market": "XSHE",
    "securityId": "000001",
    "bidPrice": 14.20,
    "askPrice": 14.25
  }
]
```

> ⚠️ **任务书勘误**：任务书 3.3 节中 `bidPrice` 标注为"最新卖价"、`askPrice` 标注为"最新买价"，与行业惯例矛盾。标准定义：**bid = 买方报价（买方愿意支付的最高价），ask = 卖方报价（卖方愿意接受的最低价）**。本项目统一采用行业标准含义。

### 5.5 交易所确认单据回报（Confirm Report）

交易所确认已接收并受理了订单。

| 字段 | 类型 | 说明 |
|------|------|------|
| `clOrderId` | char[16] | 原始订单编号 |
| `market` | char[4] | 市场 |
| `securityId` | char[6] | 股票代码 |
| `side` | char[1] | 买卖方向 |
| `qty` | uint32 | 订单数量 |
| `price` | double | 订单价格 |
| `shareHolderId` | char[10] | 股东号 |

**JSON 示例**：
```json
{
  "clOrderId": "ORD_000001",
  "market": "XSHG",
  "securityId": "600030",
  "side": "B",
  "qty": 500,
  "price": 25.50,
  "shareHolderId": "SH00001234"
}
```

### 5.6 订单非法回报（Order Reject Report）

交易所拒绝了订单，或本系统因校验/风控失败而生成。

| 字段 | 类型 | 说明 |
|------|------|------|
| `clOrderId` | char[16] | 原始订单编号 |
| `market` | char[4] | 市场 |
| `securityId` | char[6] | 股票代码 |
| `side` | char[1] | 买卖方向 |
| `qty` | uint32 | 订单数量 |
| `price` | double | 订单价格 |
| `shareHolderId` | char[10] | 股东号 |
| `rejectCode` | int32 | 拒绝码 |
| `rejectText` | char[64] | 拒绝原因说明 |

**建议的系统内部拒绝码定义**：

| rejectCode | rejectText | 触发场景 |
|------------|-----------|---------|
| 1001 | `INVALID_FIELD` | 必填字段为空 |
| 1002 | `INVALID_MARKET` | 市场代码不合法 |
| 1003 | `INVALID_SECURITY_ID` | 股票代码格式错误 |
| 1004 | `INVALID_SIDE` | 买卖方向不合法 |
| 1005 | `INVALID_QTY` | 数量不合法（≤ 0 或非整手买入） |
| 1006 | `INVALID_PRICE` | 价格不合法（≤ 0） |
| 2001 | `WASH_TRADE_DETECTED` | 检测到对敲交易 |
| 3001 | `CANCEL_ORDER_NOT_FOUND` | 待撤订单不存在 |
| 3002 | `CANCEL_ORDER_FINISHED` | 待撤订单已完成（已成交/已撤销） |

### 5.7 成交回报（Execution Report）

交易所或本系统内部撮合产生的成交结果。

| 字段 | 类型 | 说明 |
|------|------|------|
| `clOrderId` | char[16] | 原始订单编号 |
| `market` | char[4] | 市场 |
| `securityId` | char[6] | 股票代码 |
| `side` | char[1] | 买卖方向 |
| `qty` | uint32 | 原始订单数量 |
| `price` | double | 原始订单价格 |
| `shareHolderId` | char[10] | 股东号 |
| `execId` | char[12] | 成交编号（唯一） |
| `execQty` | uint32 | 本次成交数量 |
| `execPrice` | double | 本次成交价格 |

> ⚠️ **任务书勘误**：任务书 3.6 节中字段名写作 `cl0rderId`（含数字 0），疑为笔误，应为 `clOrderId`。本项目统一使用 `clOrderId`。3.7 节同理。

**JSON 示例**：
```json
{
  "clOrderId": "ORD_000001",
  "market": "XSHG",
  "securityId": "600030",
  "side": "B",
  "qty": 500,
  "price": 25.50,
  "shareHolderId": "SH00001234",
  "execId": "EXEC_000001",
  "execQty": 300,
  "execPrice": 25.45
}
```

### 5.8 撤单确认回报（Cancel Confirm Report）

| 字段 | 类型 | 说明 |
|------|------|------|
| `clOrderId` | char[16] | 撤单请求编号 |
| `origClOrderId` | char[16] | 被撤原始订单编号 |
| `market` | char[4] | 市场 |
| `securityId` | char[6] | 股票代码 |
| `shareHolderId` | char[10] | 股东号 |
| `side` | char[1] | 买卖方向 |
| `qty` | uint32 | 原始订单数量 |
| `price` | double | 原始订单价格 |
| `cumQty` | uint32 | 累计已成交数量 |

## 6. 行情数据的作用与机制

### 6.1 行情与撮合的关系

在高级目标中，行情数据对内部撮合引擎施加了额外约束：

- **内部撮合的成交价必须与当前市场行情保持合理一致**
- 不能以偏离市场太远的价格完成内部撮合，否则就是变相的价格操纵

### 6.2 行情约束规则

| 约束 | 说明 | 举例 |
|------|------|------|
| 买入价 ≤ askPrice | 内部撮合的买方成交价不应高于当前市场卖价 | 若 askPrice=25.50，买方出价 26.00，成交价应不高于 25.50 |
| 卖出价 ≥ bidPrice | 内部撮合的卖方成交价不应低于当前市场买价 | 若 bidPrice=25.45，卖方出价 25.00，成交价应不低于 25.45 |
| 成交价在 [bidPrice, askPrice] 区间内 | 成交价应落在买卖价差之内 | bidPrice=25.45, askPrice=25.50，成交价应在 [25.45, 25.50] |

### 6.3 无行情数据时的处理

在基础目标阶段（未接入行情），内部撮合的成交价可以采用简单策略：
- 取先到订单的价格（时间优先原则）
- 或取两者价格的中间值
- 具体策略在文档二中详细讨论

## 7. 任务书勘误与约定

在仔细审阅任务书后，我们发现以下疑似笔误或不一致之处，并给出本项目的统一约定：

### 7.1 买卖方向标记

| 位置 | 任务书描述 | 问题 | 本项目约定 |
|------|-----------|------|-----------|
| 3.1 节 | `B`：卖，`S`：买 | 与行业惯例相反 | **`B` = Buy = 买入，`S` = Sell = 卖出** |
| 3.6 / 3.7 节 | `B`：买，`S`：卖 | 与 3.1 节矛盾 | 同上，以 3.6/3.7 为准 |

### 7.2 行情价格含义

| 位置 | 任务书描述 | 问题 | 本项目约定 |
|------|-----------|------|-----------|
| 3.3 节 | `bidPrice`：最新卖价 | 与行业惯例相反 | **`bidPrice` = 买方报价（最优买价）** |
| 3.3 节 | `askPrice`：最新买价 | 与行业惯例相反 | **`askPrice` = 卖方报价（最优卖价）** |

### 7.3 字段命名不一致

| 位置 | 任务书写法 | 疑似问题 | 本项目约定 |
|------|-----------|---------|-----------|
| 3.6 / 3.7 节 | `cl0rderId`（含数字 0） | 可能是 `clOrderId` 的笔误 | 统一使用 **`clOrderId`** |
| 3.6 / 3.7 节 | `shareholderId`（小写 h） | 与 3.1/3.4/3.5 的 `shareHolderId` 不一致 | 统一使用 **`shareHolderId`** |
| 3.7 节 | `origCl0rderId` | 同上 | 统一使用 **`origClOrderId`** |

### 7.4 市场代码描述

| 位置 | 任务书描述 | 问题 | 本项目约定 |
|------|-----------|------|-----------|
| 3.2 / 3.3 节 | `XSHE`：上交所 | 应为深交所 | **`XSHE` = 深交所** |
