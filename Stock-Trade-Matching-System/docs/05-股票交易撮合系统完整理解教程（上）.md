## 第一章：项目对比与技术升级概览

### 1.1 你已经掌握的知识（仓库管理系统回顾）

让我们先回顾一下你的 Hello World 项目的核心结构：

```
SpringbootApplication (启动类)
    ↓
EmployeeController (接收HTTP请求)
    ↓
EmployeeService (业务逻辑)
    ↓
EmployeeMapper (数据库操作)
```

**关键特征：**
- **三层架构**：Controller → Service → Mapper
- **MyBatis 持久层**：使用 XML 或注解映射 SQL
- **简单的 CRUD**：增删改查直接对应数据库操作
- **同步返回**：每个请求处理完毕后立即返回结果
- **基础注解**：`@RestController`, `@GetMapping`, `@PostMapping`, `@PutMapping`, `@DeleteMapping`

**代码示例回顾：**

```java
@RestController
@RequestMapping("/employee")
public class EmployeeController {
    
    @Resource
    private EmployeeService employeeService;
    
    @PostMapping("/add")
    public Result add(@RequestBody Employee employee){
        employeeService.add(employee);
        return Result.success();
    }
}
```

这就是你熟悉的编程模式：收到请求 → 调用服务 → 操作数据库 → 返回结果。

### 1.2 股票交易系统的复杂性升级

股票交易撮合系统在你已有的基础上，增加了以下复杂度：

| 维度 | 仓库管理系统 | 股票交易系统 |
|------|------------|------------|
| **业务逻辑** | 简单 CRUD | 多步骤流程（校验→检测→撮合→回报） |
| **数据流向** | 单向：请求→响应 | 双向：HTTP请求 + WebSocket推送 |
| **持久层** | MyBatis + 数据库 | JPA + 内存数据结构 + 数据库 |
| **设计模式** | 简单分层 | 责任链、策略、代理等多种模式 |
| **并发处理** | 无显式并发控制 | synchronized 锁、并发安全设计 |
| **代码简化** | 手写 getter/setter | Lombok 自动生成 |
| **API 文档** | 无 | Swagger 自动生成 |

**核心数据流对比：**

仓库管理系统：
```
用户请求 → Controller → Service → Mapper → 数据库
                                              ↓
                                          返回结果
```

股票交易系统：
```
用户提交订单 → Controller → Service (多步骤处理)
                               ↓
                         [校验] → [对敲检测] → [撮合/转发]
                               ↓
                         实时回报 (WebSocket推送)
                               ↓
                         同步返回结果摘要
```

## 第二章：新技术栈详解

在深入理解股票交易系统之前，我们需要先掌握它使用的新技术和工具。

### 2.1 Lombok：告别重复代码

**你遇到的问题：**
在仓库管理系统中，`Employee` 类有大量的 getter/setter：

```java
public class Employee {
    private Integer id;
    private String name;
    
    public Integer getId() { return id; }
    public void setId(Integer id) { this.id = id; }
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
    // ... 更多字段的 getter/setter
}
```

**Lombok 的解决方案：**

```java
@Data  // 自动生成 getter、setter、toString、equals、hashCode
public class Employee {
    private Integer id;
    private String name;
}
```

**股票交易系统中的实际应用：**

让我们看看 `InternalOrder` 类（[InternalOrder.java](../backend/matching-system/src/main/java/com/stocktrade/matchingsystem/common/model/InternalOrder.java)）：

```java
@Data                    // 生成所有字段的 getter/setter
@Builder                 // 生成建造者模式代码
@NoArgsConstructor       // 生成无参构造函数
@AllArgsConstructor      // 生成全参构造函数
public class InternalOrder {
    private String clOrderId;
    private String market;
    private OrderSide side;
    private int qty;
    private double price;
    
    @Builder.Default     // 指定默认值
    private OrderStatus status = OrderStatus.RECEIVED;
}
```

**Lombok 注解详解：**

| 注解 | 作用 | 生成内容 |
|-----|------|---------|
| `@Data` | 数据类 | getter/setter/toString/equals/hashCode |
| `@Builder` | 建造者模式 | `InternalOrder.builder().clOrderId("123").build()` |
| `@NoArgsConstructor` | 无参构造 | `new InternalOrder()` |
| `@AllArgsConstructor` | 全参构造 | `new InternalOrder(id, market, ...)` |
| `@Slf4j` | 日志 | 自动创建 `log` 变量 |
| `@RequiredArgsConstructor` | 必需参数构造 | 为 `final` 和 `@NonNull` 字段生成构造函数 |

**@Builder 实战示例：**

不使用 Builder（传统方式）：
```java
InternalOrder order = new InternalOrder();
order.setClOrderId("ORD001");
order.setMarket("SH");
order.setSecurityId("600000");
order.setSide(OrderSide.BUY);
order.setQty(1000);
order.setPrice(10.5);
```

使用 Builder（链式调用）：
```java
InternalOrder order = InternalOrder.builder()
    .clOrderId("ORD001")
    .market("SH")
    .securityId("600000")
    .side(OrderSide.BUY)
    .qty(1000)
    .price(10.5)
    .build();
```

**@Slf4j 的使用：**

不使用 Lombok：
```java
public class OrderController {
    private static final Logger log = LoggerFactory.getLogger(OrderController.class);
    
    public void method() {
        log.info("Processing order");
    }
}
```

使用 Lombok：
```java
@Slf4j  // 自动创建 log 对象
public class OrderController {
    public void method() {
        log.info("Processing order");  // 直接使用
    }
}
```

### 2.2 依赖注入进阶：从 @Resource 到 @RequiredArgsConstructor

**你熟悉的方式（仓库管理系统）：**

```java
@Service
public class EmployeeService {
    @Resource  // 字段注入
    private EmployeeMapper employeeMapper;
}
```

**股票交易系统的方式（推荐的构造器注入）：**

```java
@Service
@RequiredArgsConstructor  // Lombok 自动生成构造函数
public class OrderProcessingService {
    
    private final OrderValidationChain validationChain;  // final 字段
    private final WashTradeDetector washTradeDetector;
    private final MatchingEngine matchingEngine;
    
    // Lombok 自动生成：
    // public OrderProcessingService(
    //     OrderValidationChain validationChain,
    //     WashTradeDetector washTradeDetector,
    //     MatchingEngine matchingEngine
    // ) {
    //     this.validationChain = validationChain;
    //     this.washTradeDetector = washTradeDetector;
    //     this.matchingEngine = matchingEngine;
    // }
}
```

**为什么推荐构造器注入？**
1. **不可变性**：`final` 字段保证依赖不会被修改
2. **便于测试**：可以直接 `new` 对象进行单元测试
3. **循环依赖检测**：编译期就能发现循环依赖问题

### 2.3 Swagger/OpenAPI：自动生成 API 文档

**你的现状：**
在仓库管理系统中，前端开发者需要问你："这个接口怎么调用？参数是什么？"

**Swagger 的解决方案：**
自动生成一个可交互的 API 文档网页，访问 `http://localhost:8080/swagger-ui.html` 即可看到所有接口。

**代码示例（[OrderController.java](../backend/matching-system/src/main/java/com/stocktrade/matchingsystem/gateway/controller/OrderController.java)）：**

```java
@RestController
@RequestMapping("/api/orders")
@Tag(name = "Order", description = "委托管理")  // Swagger: 接口分组
public class OrderController {
    
    @PostMapping
    @Operation(summary = "提交委托", description = "接收客户委托，经过校验、对敲检测、撮合/转发处理")
    public ResponseEntity<Map<String, Object>> submitOrder(
        @RequestBody @Valid OrderDTO orderDTO  // @Valid 触发字段校验
    ) {
        // ...
    }
}
```

**Swagger 注解说明：**
- `@Tag`: 给 Controller 分组和命名
- `@Operation`: 描述接口的功能
- 自动读取参数类型和返回值类型生成文档

### 2.4 Bean Validation：字段校验注解

**你熟悉的方式：**
```java
public void add(Employee employee) {
    if (employee.getName() == null || employee.getName().isEmpty()) {
        throw new CustomException("500", "姓名不能为空");
    }
    // ...
}
```

**股票交易系统的方式：**

在 DTO 类上添加校验注解：
```java
public class OrderDTO {
    
    @NotBlank(message = "订单编号不能为空")
    private String clOrderId;
    
    @NotBlank(message = "市场代码不能为空")
    private String market;
    
    @Positive(message = "数量必须为正数")
    private int qty;
    
    @Positive(message = "价格必须为正数")
    private double price;
}
```

在 Controller 中启用校验：
```java
@PostMapping
public ResponseEntity<?> submitOrder(
    @RequestBody @Valid OrderDTO orderDTO  // @Valid 自动触发校验
) {
    // 如果校验失败，Spring 会自动返回 400 错误
    // 如果校验通过，才会执行这里的代码
}
```

**常用校验注解：**
- `@NotNull`: 不能为 null
- `@NotBlank`: 字符串不能为空白
- `@Positive`: 必须为正数
- `@Min(10)`: 最小值
- `@Max(100)`: 最大值
- `@Size(min=1, max=10)`: 集合或字符串长度范围

打开 [pom.xml](../backend/matching-system/pom.xml)，让我们逐一理解每个依赖的作用：

```xml
<dependencies>
    <!-- Spring Boot Web: HTTP 接口支持 -->
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-web</artifactId>
    </dependency>
    
    <!-- Spring Boot WebSocket: 实时推送支持 -->
    <dependency>

    <dependency>

    </dependency>
    
    <!-- H2 Database: 内存数据库（开发测试用） -->
    <dependency>
        <groupId>com.h2database</groupId>

    </dependency>
    
    <!-- Lombok: 代码简化工具 -->
    <dependency>

        <optional>true</optional>
    </dependency>
    
    <!-- SpringDoc OpenAPI: Swagger 文档生成 -->
    <dependency>
        <groupId>org.springdoc</groupId>
        <artifactId>springdoc-openapi-starter-webmvc-ui</artifactId>
        <version>2.8.8</version>
    </dependency>
    
    <!-- Validation: 字段校验注解支持 -->
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-validation</artifactId>
    </dependency>
</dependencies>
```

**Maven 编译插件配置：**

```xml
<build>
    <plugins>
        <!-- Lombok 注解处理器配置 -->
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-compiler-plugin</artifactId>
            <configuration>
                <annotationProcessorPaths>
                    <path>
                        <groupId>org.projectlombok</groupId>
                        <artifactId>lombok</artifactId>
                    </path>
                </annotationProcessorPaths>
            </configuration>
        </plugin>
    </plugins>
</build>
```

**关键点：Lombok 的编译时处理**

Lombok 不是运行时反射生成代码，而是在**编译时**通过注解处理器生成代码：

```
.java 源文件 (含 @Data 注解)
    ↓ 
Maven 编译 (javac + Lombok 注解处理器)
    ↓
.class 字节码 (已包含 getter/setter)
    ↓
运行时 JVM 加载
```

这也是为什么你的 IDE（如 IntelliJ IDEA）需要安装 Lombok 插件，才能识别这些自动生成的方法。

### 3.2 核心数据模型

#### 3.2.1 订单数据的三种形态

**1. OrderDTO（数据传输对象）**

作用：前端提交的原始订单数据

```java
public class OrderDTO {
    private String clOrderId;    // 客户订单编号
    private String market;       // 市场代码（SH/SZ）
    private String securityId;   // 证券代码（600000）
    private String side;         // 买卖方向（B/S）
    private int qty;             // 数量
    private double price;        // 价格
    private String shareHolderId; // 股东账号
}
```

**2. InternalOrder（内部订单对象）**

作用：系统内部流转的订单，包含完整的状态信息

```java
@Data
@Builder
public class InternalOrder {
    // 原始订单字段
    private String clOrderId;
    private String market;
    private OrderSide side;      // 枚举类型（不是字符串）
    private int qty;
    private double price;
    
    // 系统状态字段
    private OrderStatus status;  // 当前状态
    private int remainQty;       // 剩余未成交数量
    private int cumQty;          // 累计已成交数量
    private boolean washTrade;   // 是否被标记为对敲
    private Instant receivedAt;  // 接收时间
    
    // 从 DTO 构建
    public static InternalOrder fromDTO(OrderDTO dto) {
        return InternalOrder.builder()
            .clOrderId(dto.getClOrderId())
            .side(OrderSide.fromCode(dto.getSide()))  // 字符串转枚举
            .remainQty(dto.getQty())  // 初始剩余量 = 总量
            .status(OrderStatus.RECEIVED)
            .build();
    }
}
```

**3. OrderEntity（数据库实体）**

作用：持久化到数据库的订单记录

```java
@Entity
@Table(name = "orders")
public class OrderEntity {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    
    private String clOrderId;
    // ...
}
```

**为什么要有三种形态？**

- **OrderDTO**: 接收外部输入，字段校验，不可信
- **InternalOrder**: 内部处理，类型安全，状态完整
- **OrderEntity**: 数据库持久化，脱离内存生命周期

### 3.4 应用启动流程

#### 3.4.1 仓库管理系统的启动（你熟悉的）

```java
@SpringBootApplication
@MapperScan("com.example.mapper")  // 扫描 MyBatis Mapper 接口
public class SpringbootApplication {
    public static void main(String[] args) {
        SpringApplication.run(SpringbootApplication.class, args);
    }
}
```

启动过程：
1. Spring Boot 扫描 `@SpringBootApplication` 所在包及子包
2. 找到所有 `@Controller`, `@Service`, `@Repository` 注解的类
3. 创建这些类的实例（Bean）
4. 处理 `@Resource` 或 `@Autowired` 注入
5. 启动 Tomcat 服务器，监听 8080 端口

#### 3.4.2 股票交易系统的启动

```java
@SpringBootApplication
public class MatchingSystemApplication {
    public static void main(String[] args) {
        SpringApplication.run(MatchingSystemApplication.class, args);
    }
}
```

看起来一样，但实际上 Spring Boot 会自动：

1. **扫描组件**：
   - `@RestController`: OrderController
   - `@Service`: OrderProcessingService, MatchingEngine, WashTradeDetector...
   - `@Configuration`: WebSocketConfig, OpenApiConfig

2. **处理依赖注入**（DI）：
   ```java
   @Service
   @RequiredArgsConstructor
   public class OrderProcessingService {
       private final WashTradeDetector washTradeDetector;  // Spring 会自动注入
   }
   ```

3. **启动 WebSocket**：
   - 读取 `WebSocketConfig`
   - 注册 `/ws` 端点
   - 启用 STOMP 协议

4. **生成 Swagger 文档**：
   - 扫描所有 `@RestController`
   - 读取 `@Operation` 注解
   - 在 `/swagger-ui.html` 提供可视化文档

5. **初始化 JPA**：
   - 读取 `@Entity` 类
   - 根据 `ddl-auto: create-drop` 自动建表

### 4.2 与仓库管理系统流程的对比

**仓库管理系统添加员工：**
```
1. Controller 接收请求
2. Service 调用 Mapper
3. Mapper 执行 INSERT SQL
4. 返回成功结果
```

**股票交易系统提交订单：**
```
1. Controller 接收请求 + @Valid 自动校验
2. Service 编排多步骤流程：
   a. 自定义业务校验（责任链模式）
   b. 对敲风控检测
   c. 根据检测结果分支：
      - 正常：转发交易所
      - 对敲：内部撮合 + 撤单
3. 异步推送 WebSocket 回报
4. 同步返回处理结果摘要
```

**关键差异：**
1. **单一责任 vs 多步骤编排**
2. **同步返回 vs 双向通信（HTTP + WebSocket）**
3. **简单校验 vs 责任链 + 风控**
4. **直接操作数据库 vs 内存管理 + 异步持久化**

## 第五章：枚举类型的深度应用

在你的仓库管理系统中，可能用字符串表示状态或类型：

```java
String role = "admin";  // 容易拼写错误，没有类型约束
if (role.equals("adimn")) { ... }  // 拼写错误，编译期无法发现
```

股票交易系统大量使用枚举类型，提供**类型安全**和**业务语义**。

### 5.1 OrderStatus 枚举（订单状态）

查看 [OrderStatus.java](../backend/matching-system/src/main/java/com/stocktrade/matchingsystem/common/constants/OrderStatus.java)：

```java
public enum OrderStatus {
    RECEIVED("已接收"),
    VALIDATED("已校验"),
    SENT("已发送至交易所"),
    REJECTED("已拒绝/拦截"),
    CONFIRMED("交易所已确认"),
    PARTIALLY_FILLED("部分成交"),
    FULLY_FILLED("全部成交"),
    CANCELED("已撤销");

    private final String description;

    OrderStatus(String description) {
        this.description = description;
    }

    public String getDescription() {
        return description;
    }

    // 业务方法：判断是否为终态
    public boolean isTerminal() {
        return this == FULLY_FILLED || this == REJECTED;
    }
}
```

**使用示例：**

```java
InternalOrder order = ...;
order.setStatus(OrderStatus.VALIDATED);  // 类型安全

if (order.getStatus() == OrderStatus.CONFIRMED) {  // 可以用 == 比较
    log.info("订单已确认");
}

if (order.getStatus().isTerminal()) {  // 调用业务方法
    log.info("订单已完结");
}
```

**对比字符串方式的优势：**
1. **编译期检查**：`OrderStatus.RECEVED` 会报错（拼写错误）
2. **IDE 自动補全**：输入 `OrderStatus.` 就能看到所有选项
3. **可以添加业务方法**：如 `isTerminal()`
4. **Switch 语句完整性检查**

### 5.2 OrderSide 枚举（买卖方向）

```java
public enum OrderSide {
    BUY("B", "买入"),
    SELL("S", "卖出");

    private final String code;
    private final String description;

    OrderSide(String code, String description) {
        this.code = code;
        this.description = description;
    }

    public String getCode() {
        return code;
    }

    // 从字符串代码转换为枚举
    public static OrderSide fromCode(String code) {
        for (OrderSide side : values()) {
            if (side.code.equals(code)) {
                return side;
            }
        }
        throw new IllegalArgumentException("Invalid side code: " + code);
    }
}
```

**为什么需要 `fromCode()` 方法？**

前端提交的 OrderDTO 中 `side` 是字符串 "B" 或 "S"，转换为内部订单时需要转为枚举：

```java
public static InternalOrder fromDTO(OrderDTO dto) {
    return InternalOrder.builder()
        .side(OrderSide.fromCode(dto.getSide()))  // "B" → OrderSide.BUY
        .build();
}
```

## 第六章：编译与运行机制

### 6.1 Maven 编译流程

当你执行 `mvn clean compile` 时，发生了什么？

```
1. Clean 阶段
   └─ 删除 target/ 目录

2. Compile 阶段
   ├─ 读取 pom.xml 依赖配置
   ├─ 下载依赖 JAR 包到本地仓库（~/.m2/repository）
   ├─ 调用 javac 编译 .java 文件
   │  └─ 同时调用 Lombok 注解处理器
   │     └─ 处理 @Data, @Builder 等注解
   │     └─ 生成对应的 .class 字节码
   ├─ 复制 resources/ 下的文件（application.yml）到 target/classes/
   └─ 生成 target/classes/ 目录结构
```

**Lombok 编译时代码生成示例：**

源文件 `InternalOrder.java`：
```java
@Data
public class InternalOrder {
    private String clOrderId;
}
```

编译后的 `InternalOrder.class`（反编译后）：
```java
public class InternalOrder {
    private String clOrderId;
    
    public String getClOrderId() {  // Lombok 生成
        return this.clOrderId;
    }
    
    public void setClOrderId(String clOrderId) {  // Lombok 生成
        this.clOrderId = clOrderId;
    }
    
    // ... 还有 toString, equals, hashCode
}
```

### 6.2 Spring Boot 运行时启动流程

当你运行 `java -jar matching-system.jar` 或在 IDE 中启动时：

```
1. 运行 main 方法
   └─ SpringApplication.run()

2. Spring Boot 自动配置
   ├─ 读取 application.yml
   ├─ 扫描 @SpringBootApplication 所在包
   │  └─ 找到所有 @Component, @Service, @Controller 类
   ├─ 创建 ApplicationContext（Spring 容器）
   └─ 实例化所有 Bean（单例模式）

3. 依赖注入
   ├─ 分析每个 Bean 的构造函数参数
   ├─ 从容器中找到对应类型的 Bean
   └─ 注入到构造函数

4. WebSocket 配置
   ├─ 执行 WebSocketConfig.configureMessageBroker()
   └─ 注册 /ws 端点

5. JPA 初始化
   ├─ 扫描 @Entity 类
   ├─ 根据 ddl-auto: create-drop 执行 CREATE TABLE
   └─ 建立 SessionFactory

6. 启动 Tomcat
   ├─ 监听 8080 端口
   └─ 准备接收 HTTP 请求

7. 应用就绪
   └─ 日志: Started MatchingSystemApplication in X.XX seconds
```

### 6.3 一次 HTTP 请求的完整生命周期

用户提交订单：`POST http://localhost:8080/api/orders`

```
1. Tomcat 接收 HTTP 请求
   └─ 解析 JSON 请求体为 OrderDTO 对象

2. Spring MVC 前置处理
   ├─ 找到匹配的 Controller 方法（@PostMapping）
   ├─ 参数绑定：将 JSON 转为 OrderDTO
   └─ 执行 @Valid 字段校验
       ├─ 检查 @NotBlank, @Positive 等注解
       └─ 失败 → 返回 400 Bad Request

3. 进入 Controller 方法
   OrderController.submitOrder(orderDTO)

4. 调用 Service 层
   OrderProcessingService.processOrder(orderDTO)
   └─ 执行业务逻辑（详见第四章流程图）

5. 构造 HTTP 响应
   ├─ 将返回值转为 JSON
   └─ 设置 Content-Type: application/json

6. Tomcat 发送响应
   └─ 前端收到 200 OK

7. 异步推送 WebSocket（并行）
   └─ ReportDispatcher 推送实时回报
```
