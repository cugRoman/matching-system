## 第七章：订单校验 — 责任链模式详解

### 7.1 什么是责任链模式？

**你熟悉的方式（仓库管理系统）：**

```java
public void add(Employee employee) {
    // 一个方法里堆砌所有校验逻辑
    if (employee.getName() == null || employee.getName().isEmpty()) {
        throw new CustomException("500", "姓名不能为空");
    }
    if (employee.getAge() != null && employee.getAge() < 18) {
        throw new CustomException("500", "年龄必须大于18岁");
    }
    if (employee.getNo() == null) {
        throw new CustomException("500", "工号不能为空");
    }
    // ... 更多校验
    // 真正的业务逻辑
    employeeMapper.insert(employee);
}
```

**问题：**
1. 校验逻辑和业务逻辑混在一起
2. 难以扩展（新增校验需要修改原方法）
3. 无法复用（其他地方也需要这些校验就得复制粘贴）

**责任链模式的解决方案：**

将每个校验规则独立成一个类，通过链式调用依次执行：

```
订单 → Rule1 → Rule2 → Rule3 → ... → 全部通过 / 某个拒绝
```

### 7.2 股票交易系统的校验架构

**核心类：ValidationRule 接口**

```java
public interface ValidationRule {
    /**
     * 校验订单
     * @return empty = 通过; present = 失败的拒绝回报
     */
    Optional<RejectReportDTO> validate(OrderDTO order);
}
```

**具体校验规则实现（以 RequiredFieldRule 为例）：**

查看 [RequiredFieldRule.java](../backend/matching-system/src/main/java/com/stocktrade/matchingsystem/validator/rules/RequiredFieldRule.java)：

```java
@Component
@Order(1)  // 执行顺序：1 最先执行
public class RequiredFieldRule implements ValidationRule {

    @Override
    public Optional<RejectReportDTO> validate(OrderDTO order) {
        if (order.getClOrderId() == null || order.getClOrderId().isBlank()) {
            return Optional.of(buildReject(order, RejectCode.INVALID_FIELD, 
                "clOrderId is required"));
        }
        if (order.getShareHolderId() == null || order.getShareHolderId().isBlank()) {
            return Optional.of(buildReject(order, RejectCode.INVALID_FIELD, 
                "shareHolderId is required"));
        }
        return Optional.empty();  // 通过
    }
    
    private RejectReportDTO buildReject(OrderDTO order, RejectCode code, String detail) {
        return RejectReportDTO.builder()
            .clOrderId(order.getClOrderId())
            .rejectCode(code.getCode())
            .rejectText(detail)
            .build();
    }
}
```

**关键点解析：**

1. **`@Component`**: 让 Spring 自动扫描并创建这个 Bean
2. **`@Order(1)`**: 指定执行顺序（数字越小越先执行）
3. **`Optional<RejectReportDTO>`**: 
   - `Optional.empty()` 表示校验通过
   - `Optional.of(reject)` 表示校验失败，返回拒绝理由

**其他校验规则示例：**

```java
@Component
@Order(2)
public class PriceRule implements ValidationRule {
    @Override
    public Optional<RejectReportDTO> validate(OrderDTO order) {
        if (order.getPrice() <= 0) {
            return Optional.of(buildReject(..., "价格必须大于0"));
        }
        return Optional.empty();
    }
}

@Component
@Order(3)
public class QuantityRule implements ValidationRule {
    @Override
    public Optional<RejectReportDTO> validate(OrderDTO order) {
        if (order.getQty() <= 0 || order.getQty() % 100 != 0) {
            return Optional.of(buildReject(..., "数量必须是100的整数倍"));
        }
        return Optional.empty();
    }
}
```

**责任链编排器：OrderValidationChain**

查看 [OrderValidationChain.java](../backend/matching-system/src/main/java/com/stocktrade/matchingsystem/validator/OrderValidationChain.java)：

```java
@Component
public class OrderValidationChain {

    private final List<ValidationRule> rules;

    // Spring 会自动注入所有实现了 ValidationRule 接口的 Bean
    // 并按照 @Order 注解排序
    public OrderValidationChain(List<ValidationRule> rules) {
        this.rules = rules;
    }

    public Optional<RejectReportDTO> validate(OrderDTO order) {
        for (ValidationRule rule : rules) {
            Optional<RejectReportDTO> result = rule.validate(order);
            if (result.isPresent()) {
                return result;  // 短路：第一个失败就返回
            }
        }
        return Optional.empty();  // 全部通过
    }
}
```

**Spring 依赖注入的魔法：**

当你写 `List<ValidationRule> rules` 作为构造函数参数时，Spring 会：
1. 找到所有标注了 `@Component` 且实现了 `ValidationRule` 的类
2. 按照 `@Order` 注解排序
3. 创建实例并注入到这个 List 中

```java
rules = [
    RequiredFieldRule (Order=1),
    PriceRule (Order=2),
    QuantityRule (Order=3),
    MarketRule (Order=4),
    ...
]
```

### 7.3 责任链模式的优势

对比仓库管理系统的方式：

**扩展性：**
- 旧方式：修改原方法，加 if 语句
- 责任链：新建一个 `XxxRule` 类，加 `@Component` 和 `@Order`，自动生效

**复用性：**
- 旧方式：复制粘贴校验代码
- 责任链：校验链可以被多个 Service 共享使用

**可测试性：**
- 旧方式：测试整个方法，难以单独测试某个校验逻辑
- 责任链：每个 Rule 独立测试

**示例：新增一个校验规则**

需求：价格不能大于涨停价（前收盘价 * 1.1）

```java
@Component
@Order(10)  // 在其他基础校验之后执行
public class PriceLimitRule implements ValidationRule {
    
    @Autowired
    private MarketDataService marketDataService;  // 获取行情数据
    
    @Override
    public Optional<RejectReportDTO> validate(OrderDTO order) {
        double prevClose = marketDataService.getPrevClose(
            order.getMarket(), order.getSecurityId()
        );
        double upperLimit = prevClose * 1.1;
        
        if (order.getPrice() > upperLimit) {
            return Optional.of(buildReject(order, 
                "价格超过涨停价: " + upperLimit));
        }
        return Optional.empty();
    }
}
```

**不需要修改任何现有代码！**Spring 会自动把这个新规则加入到责任链中。

## 第八章：对敲检测 — 并发安全设计

### 8.1 什么是对敲交易？

**业务场景：**
股东 A 同时挂出：
- 买单：买入 工商银行(601398) 1000 股 @ 5.00 元
- 卖单：卖出 工商银行(601398) 1000 股 @ 4.95 元

这两笔订单可以互相成交，但这是**同一个人左手倒右手**，可能涉及洗钱、操纵股价等违法行为，需要拦截。

**对敲检测条件：**
1. 同一股东账号（`shareHolderId`）
2. 同一市场（`market`）
3. 同一证券（`securityId`）
4. 买卖方向相反（一个买、一个卖）
5. 价格可成交（买价 >= 卖价）

### 8.2 核心数据结构：OrderIndex

查看 [OrderIndex.java](../backend/matching-system/src/main/java/com/stocktrade/matchingsystem/risk/OrderIndex.java)：

**索引结构设计：**

```java
ConcurrentHashMap<
    String,  // washTradeKey = "股东ID|市场|证券代码"
    Map<
        OrderSide,  // BUY / SELL
        TreeMap<
            Double,  // 价格（排序）
            Deque<InternalOrder>  // 同价位的订单队列（时间优先）
        >
    >
>
```

**示例数据：**

```
Key: "A123456|SH|600000"
Value: {
    BUY: {
        10.50 -> [订单1, 订单2],  // 2笔买单 @ 10.50
        10.48 -> [订单3]           // 1笔买单 @ 10.48
    },
    SELL: {
        10.52 -> [订单4],
        10.55 -> [订单5, 订单6]
    }
}
```

**为什么用 TreeMap？**

TreeMap 的 key（价格）自动排序：
- 买方：从高到低（优先匹配高价买单）
- 卖方：从低到高（优先匹配低价卖单）

**为什么用 Deque？**

同一价格的多个订单，按时间优先原则（先到先配对）。

### 8.3 查找对手方订单

```java
public List<InternalOrder> findCounterOrders(InternalOrder incoming) {
    String key = incoming.washTradeKey();  // "股东|市场|证券"
    Map<OrderSide, TreeMap<...>> sideMap = index.get(key);
    
    OrderSide counterSide = incoming.getSide().opposite();  // 对手方向
    TreeMap<Double, Deque<InternalOrder>> counterBook = sideMap.get(counterSide);
    
    List<InternalOrder> result = new ArrayList<>();
    
    if (incoming.getSide() == OrderSide.BUY) {
        // 买单到来，找卖价 <= 买价的卖单
        NavigableMap<...> eligible = counterBook.headMap(incoming.getPrice(), true);
        for (Deque<InternalOrder> queue : eligible.values()) {
            result.addAll(queue);
        }
    } else {
        // 卖单到来，找买价 >= 卖价的买单
        NavigableMap<...> eligible = counterBook.tailMap(incoming.getPrice(), true);
        for (Deque<InternalOrder> queue : eligible.descendingMap().values()) {
            result.addAll(queue);
        }
    }
    
    return result;
}
```

**实际案例分析：**

已有订单索引：
```
"A123|SH|600000":
    BUY: { 10.50 -> [订单A], 10.30 -> [订单B] }
    SELL: { 10.40 -> [订单C] }
```

新来一笔卖单：`A123 卖出 600000 @ 10.45`

检测过程：
1. 查找 BUY 方向的订单
2. 找买价 >= 10.45 的：只有 10.50
3. 返回 `[订单A]`

结论：**检测到对敲！**同一股东的买卖单价格可成交。

### 8.4 并发安全：为什么需要锁？

**问题场景：**

```
时间轴：
t1: 订单A（买）进入系统，开始检测
t2: 订单B（卖）进入系统，开始检测  ← 并发！
t3: 订单A 检测完毕，未发现对手方，转发交易所
t4: 订单B 检测完毕，未发现对手方，转发交易所
```

**结果：**两笔对敲订单都漏掉了！因为检测时对方还没加入索引。

**解决方案：同一 washTradeKey 的订单串行处理**

```java
// OrderIndex 提供锁对象
private final ConcurrentHashMap<String, Object> locks = new ConcurrentHashMap<>();

public Object getLock(String washTradeKey) {
    return locks.computeIfAbsent(washTradeKey, k -> new Object());
}
```

**OrderProcessingService 中的加锁：**

```java
String washKey = internalOrder.washTradeKey();  // "股东|市场|证券"

synchronized (orderIndex.getLock(washKey)) {  // 获取锁
    
    // 1. 对敲检测
    WashTradeResult washResult = washTradeDetector.detect(internalOrder);
    
    if (!washResult.isDetected()) {
        // 2. 加入索引
        orderIndex.addOrder(internalOrder);
        
        // 3. 转发交易所
        exchangeProxy.forwardOrder(orderDTO);
    } else {
        // 对敲处理逻辑
    }
    
}  // 释放锁
```

**锁的粒度：**

- ❌ **不能锁全局**：会导致所有订单都串行，性能极差
- ✅ **锁 washTradeKey**：只有同一股东+同一股票的订单互斥，其他订单并行处理

```
股东A 工商银行  ←---|
股东A 招商银行  ←---+  不同 washTradeKey，并行处理
股东B 工商银行  ←---|

股东A 工商银行 买单 ←---|
股东A 工商银行 卖单 ←---+  相同 washTradeKey，串行处理（加锁）
```

## 第九章：内部撮合引擎

### 9.1 撮合是什么？

**简单理解：**
买方想买，卖方想卖，价格合适就配对成交。

**示例：**
- 订单1：买入 600000 @ 10.50，数量 1000
- 订单2：卖出 600000 @ 10.45，数量 500

配对结果：
- 成交 500 股 @ 10.45（取先到订单的价格）
- 订单1 剩余 500 股未成交
- 订单2 全部成交

### 9.2 撮合引擎实现

查看 [MatchingEngine.java](../backend/matching-system/src/main/java/com/stocktrade/matchingsystem/matching/MatchingEngine.java)：

```java
@Service
@RequiredArgsConstructor
public class MatchingEngine {

    private final OrderIndex orderIndex;

    public MatchResult match(InternalOrder incoming, List<InternalOrder> counterOrders) {
        List<ExecReportDTO> reports = new ArrayList<>();
        int remainQty = incoming.getRemainQty();  // 待成交数量
        
        for (InternalOrder counter : counterOrders) {
            if (remainQty <= 0) break;  // incoming 已全部成交
            if (counter.getRemainQty() <= 0) continue;  // counter 已成交完
            
            // 成交数量 = min(incoming剩余, counter剩余)
            int execQty = Math.min(remainQty, counter.getRemainQty());
            
            // 成交价 = 先到订单的价格（被动方报价原则）
            double execPrice = counter.getPrice();
            
            // 生成唯一成交编号
            String execId = IdGenerator.nextExecId();
            
            // 确定买卖双方
            InternalOrder buyOrder = incoming.getSide() == OrderSide.BUY 
                ? incoming : counter;
            InternalOrder sellOrder = incoming.getSide() == OrderSide.BUY 
                ? counter : incoming;
            
            // 生成买卖双方的成交回报
            reports.add(buildExecReport(buyOrder, execId, execQty, execPrice));
            reports.add(buildExecReport(sellOrder, execId, execQty, execPrice));
            
            // 更新剩余数量
            remainQty -= execQty;
            incoming.setRemainQty(remainQty);
            incoming.setCumQty(incoming.getCumQty() + execQty);
            
            counter.setRemainQty(counter.getRemainQty() - execQty);
            counter.setCumQty(counter.getCumQty() + execQty);
            
            // 若 counter 全部成交，从索引移除
            if (counter.getRemainQty() <= 0) {
                orderIndex.removeOrder(counter);
            }
        }
        
        return new MatchResult(reports, remainQty);
    }
}
```

**关键点解析：**

1. **成交价格原则**：`execPrice = counter.getPrice()`
   - counter 是先到的订单（挂在索引中的）
   - incoming 是后到的订单（撮合触发方）
   - 遵循"被动方报价"原则：先挂单的人的价格优先

2. **成交数量计算**：`Math.min(remainQty, counter.getRemainQty())`
   - 两边都有剩余量，取小的
   - 确保不会成交超过任何一方的量

3. **一笔成交两份回报**：
   - 买方收到一份
   - 卖方收到一份
   - 但 `execId` 相同，表示是同一笔成交

4. **累计成交量**：`cumQty`
   - 记录订单从提交到现在总共成交了多少
   - 用于前端展示"已成交/总量"

## 第十章：WebSocket 实时推送机制

### 10.1 为什么需要 WebSocket？

**你熟悉的 HTTP 模式：**

```
前端 → HTTP POST → 后端 → 返回结果 → 前端
```

问题：前端只能在发请求时得到响应，无法主动接收后端推送的消息。

**股票交易场景：**
- 用户提交订单后，成交回报可能在几秒后才到达
- 后端需要**主动推送**成交消息给前端
- 不能让前端每秒轮询一次（太低效）

**WebSocket 解决方案：**

```
前端 ←→ WebSocket 长连接 ←→ 后端
```

建立连接后，双方都可以随时发送消息。

### 10.2 WebSocket 配置

查看 [WebSocketConfig.java](../backend/matching-system/src/main/java/com/stocktrade/matchingsystem/config/WebSocketConfig.java)：

```java
@Configuration
@EnableWebSocketMessageBroker  // 启用 WebSocket
public class WebSocketConfig implements WebSocketMessageBrokerConfigurer {

    @Override
    public void configureMessageBroker(MessageBrokerRegistry config) {
        // 启用简单的内存消息代理
        config.enableSimpleBroker("/topic");  // 前端订阅的主题前缀
        
        // 前端发送消息到服务端的前缀
        config.setApplicationDestinationPrefixes("/app");
    }

    @Override
    public void registerStompEndpoints(StompEndpointRegistry registry) {
        // WebSocket 连接端点
        registry.addEndpoint("/ws")          // 前端连接 ws://localhost:8080/ws
                .setAllowedOriginPatterns("*")  // 允许跨域
                .withSockJS();               // 支持 SockJS（WebSocket 降级方案）
    }
}
```

**STOMP 协议：**
- WebSocket 只定义了如何建立连接和传输字节流
- STOMP 在 WebSocket 之上定义了消息格式（类似 HTTP 之于 TCP）
- Spring Boot 内置 STOMP 支持

### 10.3 消息发布器

查看 [WebSocketPublisher.java](../backend/matching-system/src/main/java/com/stocktrade/matchingsystem/report/WebSocketPublisher.java)：

```java
@Component
@RequiredArgsConstructor
public class WebSocketPublisher {

    private final SimpMessagingTemplate messagingTemplate;

    public void publishExecution(ExecReportDTO report) {
        messagingTemplate.convertAndSend(
            "/topic/reports/execution",  // 主题
            report                       // 消息内容（自动转 JSON）
        );
        log.debug("Published execution report: {}", report.getExecId());
    }
    
    public void publishReject(RejectReportDTO report) {
        messagingTemplate.convertAndSend("/topic/reports/reject", report);
    }
    
    public void publishConfirm(ConfirmReportDTO report) {
        messagingTemplate.convertAndSend("/topic/reports/confirm", report);
    }
}
```

**`SimpMessagingTemplate` 是什么？**

Spring 提供的 WebSocket 消息发送工具，类似于仓库管理系统中的 `HttpServletResponse`。

### 10.4 前端订阅流程（TypeScript）

前端代码示例（位于 frontend 目录）：

```typescript
import SockJS from 'sockjs-client';
import { Stomp } from '@stomp/stompjs';

// 1. 建立 WebSocket 连接
const socket = new SockJS('http://localhost:8080/ws');
const stompClient = Stomp.over(socket);

// 2. 连接服务器
stompClient.connect({}, () => {
    console.log('WebSocket connected');
    
    // 3. 订阅成交回报主题
    stompClient.subscribe('/topic/reports/execution', (message) => {
        const report = JSON.parse(message.body);
        console.log('收到成交回报:', report);
        // 更新 UI
        updateExecutionList(report);
    });
    
    // 订阅拒绝回报
    stompClient.subscribe('/topic/reports/reject', (message) => {
        const report = JSON.parse(message.body);
        alert(`订单被拒绝: ${report.rejectText}`);
    });
});
```

**消息流向：**

```
后端 OrderProcessingService
    ↓
ReportDispatcher.dispatch(result)
    ↓
WebSocketPublisher.publishExecution(report)
    ↓
SimpMessagingTemplate 推送到 /topic/reports/execution
    ↓
前端 WebSocket 收到消息
    ↓
执行回调函数 updateExecutionList(report)
```

### 10.5 HTTP vs WebSocket 对比

**仓库管理系统（HTTP）：**
```
前端提交表单 → 后端处理 → 立即返回结果 → 前端刷新列表
```

**股票交易系统（HTTP + WebSocket）：**
```
前端提交订单（HTTP）
    ↓
后端返回：{"message": "订单已接收"}（HTTP 响应）
    ↓
后端处理订单（异步）
    ↓
后端推送成交回报（WebSocket）
    ↓
前端收到推送，更新界面
```

**对比：**

| 维度 | HTTP | WebSocket |
|------|------|-----------|
| 连接 | 短连接（一次请求一次连接） | 长连接（保持打开） |
| 方向 | 单向（请求→响应） | 双向（随时互发） |
| 延迟 | 较高（需要重新建立连接） | 低（连接已建立） |
| 使用场景 | 同步操作（增删改查） | 实时推送（行情、成交） |

## 第十一章：交易所代理层

### 11.1 为什么需要 ExchangeProxy？

**业务背景：**
- 正常订单：转发到真实交易所（上交所/深交所）
- 对敲订单：拦截后内部撮合，如果对手方已发送到交易所，需要撤单

**ExchangeProxy 的职责：**
1. 转发订单到交易所
2. 记录已发送的订单
3. 发送撤单请求

### 11.2 模拟交易所

查看 [ExchangeSimulator.java](../backend/matching-system/src/main/java/com/stocktrade/matchingsystem/exchange/ExchangeSimulator.java)（简化版）：

```java
@Component
public class ExchangeSimulator {
    
    /**
     * 接收订单，返回确认回报
     */
    public ConfirmReportDTO receiveOrder(OrderDTO order) {
        log.info("Exchange received order: {}", order.getClOrderId());
        
        return ConfirmReportDTO.builder()
            .clOrderId(order.getClOrderId())
            .market(order.getMarket())
            .securityId(order.getSecurityId())
            .confirmText("已接受")
            .build();
    }
    
    /**
     * 接收撤单请求
     */
    public CancelConfirmDTO receiveCancelRequest(CancelRequestDTO req, OrderDTO original) {
        log.info("Exchange received cancel request for {}", req.getOrigClientId());
        
        return CancelConfirmDTO.builder()
            .origClOrderId(req.getOrigClientId())
            .cancelStatus("CANCELED")
            .build();
    }
}
```

**实际生产环境：**
- `ExchangeSimulator` 会被替换为真实的交易所接口调用
- 使用 FIX 协议、WebSocket 或专有协议与交易所通信

### 11.3 已发送订单注册表

```java
@Service
@RequiredArgsConstructor
public class ExchangeProxy {

    private final ExchangeSimulator exchangeSimulator;
    
    // 记录已发送到交易所的订单
    private final ConcurrentHashMap<String, OrderDTO> sentOrders = new ConcurrentHashMap<>();

    public ConfirmReportDTO forwardOrder(OrderDTO order) {
        sentOrders.put(order.getClOrderId(), order);  // 记录
        return exchangeSimulator.receiveOrder(order);
    }

    public boolean isOrderSent(String clOrderId) {
        return sentOrders.containsKey(clOrderId);
    }
}
```

**为什么需要注册表？**

对敲检测时，对手方订单可能已经发送到交易所：

```
时间轴：
t1: 买单A 未检测到对敲，转发交易所，加入 sentOrders
t2: 卖单B 到达，检测到对敲（与A配对）
t3: 需要撤回 买单A → 查询 sentOrders，发现已发送 → 发撤单请求
```

### 11.4 撤单流程

```java
public CancelConfirmDTO sendCancelRequest(String origOrderId, String cancelId) {
    OrderDTO original = sentOrders.get(origOrderId);
    if (original == null) {
        log.warn("Cannot cancel: order {} not found", origOrderId);
        return null;
    }
    
    CancelRequestDTO cancelReq = CancelRequestDTO.builder()
        .orderId(cancelId)  // 新的撤单编号
        .origClientId(origOrderId)  // 原订单编号
        .market(original.getMarket())
        .securityId(original.getSecurityId())
        .build();
    
    CancelConfirmDTO result = exchangeSimulator.receiveCancelRequest(cancelReq, original);
    
    sentOrders.remove(origOrderId);  // 从注册表移除
    return result;
}
```

```java
public void dispatch(OrderProcessResult result) {
    switch (result.getType()) {
        case REJECTED -> {
            // 发送拒绝回报
            webSocketPublisher.publishReject(result.getRejectReport());
        }
        case FORWARDED -> {
            // 发送确认回报
            webSocketPublisher.publishConfirm(result.getConfirmReport());
        }
        case WASH_TRADE_HANDLED -> {

            result.getExecReports().forEach(

            );
            
            // 3. 发送撤单确认
            result.getCancelConfirms().forEach(
                webSocketPublisher::publishCancelConfirm
            );
        }
    }
}
```

## 第十四章：代码组织的设计模式总结

### 14.1 责任链模式（Chain of Responsibility）

**应用场景：**订单校验

**角色：**
- 抽象处理者：`ValidationRule` 接口
- 具体处理者：`RequiredFieldRule`, `PriceRule`, `QuantityRule`...
- 链管理者：`OrderValidationChain`

**优势：**解耦、可扩展、可复用

### 14.2 策略模式（Strategy）

**应用场景：**不同的处理分支（转发 vs 内部撮合）

虽然代码中没有显式定义 Strategy 接口，但 `OrderProcessingService` 中的分支逻辑本质上是策略选择：

```java
if (!washResult.isDetected()) {
    // 策略1：转发交易所
    exchangeProxy.forwardOrder(orderDTO);
} else {
    // 策略2：内部撮合
    matchingEngine.match(...);
}
```

### 14.3 代理模式（Proxy）

**应用场景：**`ExchangeProxy` 代理真实交易所

**角色：**
- 抽象主题：订单转发接口
- 真实主题：`ExchangeSimulator`（模拟）或真实交易所 API
- 代理：`ExchangeProxy`（增加了订单注册、撤单管理等功能）

### 14.4 观察者模式（Observer）

**应用场景：**WebSocket 推送

**角色：**
- 主题（Subject）：后端服务
- 观察者（Observer）：前端 WebSocket 客户端
- 通知机制：`SimpMessagingTemplate`

当订单状态变化时，自动通知所有订阅了相应主题的前端客户端。

### 14.5 建造者模式（Builder）

**应用场景：**复杂对象创建（借助 Lombok）

```java
InternalOrder order = InternalOrder.builder()
    .clOrderId("ORD001")
    .market("SH")
    .securityId("600000")
    .side(OrderSide.BUY)
    .qty(1000)
    .price(10.50)
    .build();
```

**优势：**
- 可读性强（明确每个字段）
- 可选参数（不需要的字段可以不设置）
- 不可变对象（Builder 可以配合 final 字段）

### 15.6 扩展示例：新增一个校验规则

需求：限制每个股东每天最多提交100笔订单。

**步骤：**

1. 创建 `DailyOrderLimitRule.java`：

```java
package com.stocktrade.matchingsystem.validator.rules;

import com.stocktrade.matchingsystem.common.constants.RejectCode;
import com.stocktrade.matchingsystem.common.model.dto.OrderDTO;
import com.stocktrade.matchingsystem.common.model.dto.RejectReportDTO;
import com.stocktrade.matchingsystem.validator.ValidationRule;
import org.springframework.core.annotation.Order;
import org.springframework.stereotype.Component;

import java.util.Map;
import java.util.concurrent.ConcurrentHashMap;
import java.util.concurrent.atomic.AtomicInteger;

@Component
@Order(100)  // 在其他基础校验之后执行
public class DailyOrderLimitRule implements ValidationRule {

    // 记录每个股东今天的订单数量
    private final Map<String, AtomicInteger> dailyCounters = new ConcurrentHashMap<>();
    
    private static final int MAX_ORDERS_PER_DAY = 100;

    @Override
    public Optional<RejectReportDTO> validate(OrderDTO order) {
        String shareHolderId = order.getShareHolderId();
        
        AtomicInteger counter = dailyCounters.computeIfAbsent(
            shareHolderId, 
            k -> new AtomicInteger(0)
        );
        
        int count = counter.incrementAndGet();
        
        if (count > MAX_ORDERS_PER_DAY) {
            return Optional.of(RejectReportDTO.builder()
                .clOrderId(order.getClOrderId())
                .rejectCode(RejectCode.INVALID_REQUEST.getCode())
                .rejectText("超过每日订单数量限制（" + MAX_ORDERS_PER_DAY + "笔）")
                .build());
        }
        
        return Optional.empty();
    }
}
```

2. **不需要修改任何其他代码！**

Spring 会自动：
- 扫描到 `@Component`
- 识别 `implements ValidationRule`
- 按 `@Order(100)` 加入责任链
- 在每次校验时调用

3. 重启应用，测试：

```json

    "rejectCode": "INVALID_REQUEST",
    "rejectText": "超过每日订单数量限制（100笔）"
}
```

| 维度 | 仓库管理系统 | 股票交易系统 |
|------|------------|------------|
| **业务复杂度** | CRUD（直接操作数据库） | 多步骤流水线（校验→检测→撮合→回报） |
| **并发处理** | 单线程逻辑 | 多线程并发+锁控制 |
| **数据结构** | 简单实体类 | 内存索引+数据库 |
| **通信方式** | HTTP 请求-响应 | HTTP + WebSocket 双向 |
| **异常处理** | 统一异常捕获 | 多层校验+拒绝回报 |
| **可扩展性** | 修改代码 | 插件化（责任链、策略） |

### 16.2 技术栈对比

| 组件 | 仓库管理系统 | 股票交易系统 |
|------|------------|------------|
| **持久层** | MyBatis | JPA/Hibernate |
| **数据库** | MySQL | H2（可切换 PostgreSQL） |
| **代码简化** | 手写 | Lombok |
| **API 文档** | 无 | Swagger |
| **实时通信** | 无 | WebSocket (STOMP) |
| **校验** | 业务代码中堆砌 | Bean Validation + 责任链 |

### 16.3 架构思想对比

**仓库管理系统：面向过程**
```
Controller → Service → Mapper → 数据库
```
每一层直接调用下一层，紧耦合。

**股票交易系统：面向对象+设计模式**
```
Controller → Service (编排器)
                ↓
           [责任链] Validator
                ↓
           [策略] 对敲检测 → [分支1] 转发 / [分支2] 撮合
                ↓
           [代理] ExchangeProxy
                ↓
           [观察者] WebSocket 推送
```
每个模块职责单一，松耦合，通过接口和模式连接。
