## 1. 正确性难题

正确性是交易系统的核心要求。一笔错误的撮合或遗漏的风控判定，在真实场景中可能导致巨额损失或监管处罚。

### 1.1 对敲检测的精确性

#### 核心问题

对敲检测的难点不在于"规则是什么"，而在于**边界条件的精确定义和高效执行**。

#### 判定条件明确化

对敲成立的完整条件链：

```
同一 shareHolderId
  AND 同一 market
    AND 同一 securityId
      AND 方向相反（一 B 一 S）
        AND 价格可成交（买价 ≥ 卖价）
```

**每一个条件都需要正确实现，任一条件缺失都会过度拦截或遗漏。**

#### 边界场景枚举

| # | 场景描述 | 是否对敲 | 分析 |
|---|---------|---------|------|
| 1 | 同股东号，同股票，反向，买价 > 卖价 | ✅ | 标准对敲 |
| 2 | 同股东号，同股票，反向，买价 = 卖价 | ✅ | 价格相等也可成交 |
| 3 | 同股东号，同股票，反向，买价 < 卖价 | ❌ | 无法成交，不构成对敲 |
| 4 | 同股东号，**不同**股票，反向 | ❌ | 不同标的 |
| 5 | **不同**股东号，同股票，反向 | ❌ | 不同控制人，正常交易 |
| 6 | 同股东号，同股票，**同方向** | ❌ | 无反向配对 |
| 7 | 同股东号，同股票代码，**不同市场** | ❌ | 不同市场的同名股票是不同标的 |
| 8 | 同股东号，同股票，反向，先到订单已全部成交 | ❌ | 先到订单已不在簿中，无法配对 |
| 9 | 同股东号，同股票，反向，先到订单已部分成交 | ⚠️ | 需与剩余数量配对 |

#### 场景 9 的详细分析

```
T1: 买入 600030，500 股 @ 25.50，股东号 SH001  → 转发到交易所
T2: 交易所回报：部分成交 200 股                    → 订单剩余 300 股
T3: 卖出 600030，400 股 @ 25.40，股东号 SH001  → 对敲检测
```

此时对敲判定应基于**剩余未成交数量**（300 股），而非原始数量（500 股）。内部撮合最多成交 300 股（取买方剩余 300 与卖方 400 的较小值），卖方剩余 100 股需要决策：
- 方案 A：继续以正常订单转发到交易所（100 股已无法与 SH001 的买单构成对敲）
- 方案 B：整体拦截卖方订单（虽更保守，但可能不符合客户意图）

**推荐方案 A**：仅拦截能构成对敲的部分，剩余部分正常处理。

#### 多笔订单的配对选择

同一股东号下可能存在多笔同方向挂单：

```
已有挂单（同股东号 SH001，同股票 600030）:
  买1: 500 股 @ 25.50  (T1 时刻到达)
  买2: 300 股 @ 25.55  (T2 时刻到达)

新到订单:
  卖: 400 股 @ 25.45  (T3 时刻到达)
```

买1 和 买2 都与卖单构成对敲（买价 ≥ 卖价）。应该与哪一笔配对？

**遵循"价格优先、时间优先"原则**：
1. 价格优先：买2 出价 25.55 更高 → 优先
2. 若价格相同，时间优先：先到的优先

因此应先与买2（25.55 × 300 股）配对，成交 300 股。卖方剩余 100 股再与买1 配对，成交 100 股。

### 1.2 订单状态管理

#### 状态机设计

每笔订单在系统中有明确的生命周期状态：

```
                              ┌ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ┐
                                                             
                              │  ┌───────────────────────┐   │
        ┌──────────┐            │ PartiallyFilled (部分成交)│
        │ Received │──────────│─┤                       │   │
        │ (已接收)  │             └──────────┬────────────┘   │
        └────┬─────┘          │              │               │
             │                        ┌──────▼────────────┐  │
             │                │       │  FullyFilled      │  │
             │                        │  (全部成交)        │  │
       ┌─────▼─────┐         │       └───────────────────┘  │
       │ Validated  │                                        │
       │ (已校验)   │         │  以上状态由交易所 or 内部撮合驱动│
       └─────┬─────┘          └ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ┘
             │
        ┌────┴───────────────┐
        │                    │
   ┌────▼─────┐    ┌────────▼────────┐
   │  Sent    │    │   Rejected      │
   │ (已发送)  │    │  (已拒绝/拦截)   │
   └────┬─────┘    └─────────────────┘
        │
   ┌────▼───────────────┐
   │   Confirmed        │
   │  (交易所已确认)      │
   └────┬───────────────┘
        │
   ┌────┴────────────────────┐
   │                         │
┌──▼────────────┐   ┌───────▼────────┐
│PartiallyFilled│   │  FullyFilled   │
│  (部分成交)    │   │  (全部成交)     │
└──┬────────────┘   └────────────────┘
   │
┌──▼────────────┐
│  Canceled     │
│  (已撤销)      │
└───────────────┘
```

#### 状态转换规则

| 当前状态 | 允许的转换 | 触发条件 |
|---------|----------|---------|
| Received → Validated | 校验通过 | 通过基本校验 |
| Received → Rejected | 校验失败 | 字段非法 |
| Validated → Sent | 非对敲，转发 | 对敲检测未命中 |
| Validated → Rejected | 对敲拦截 | 对敲检测命中 |
| Validated → PartiallyFilled | 对敲内部撮合（部分） | 内部撮合数量 < 订单数量 |
| Validated → FullyFilled | 对敲内部撮合（全部） | 内部撮合数量 = 订单数量 |
| Sent → Confirmed | 交易所确认 | 收到确认回报 |
| Confirmed → PartiallyFilled | 交易所部分成交 | 收到成交回报，剩余 > 0 |
| Confirmed → FullyFilled | 交易所全部成交 | 收到成交回报，剩余 = 0 |
| Confirmed → Canceled | 撤单成功 | 收到撤单确认回报 |
| PartiallyFilled → FullyFilled | 继续成交 | 累计成交 = 原始数量 |
| PartiallyFilled → Canceled | 部分成交后撤单 | 剩余部分被撤销 |

#### 关键约束
- **不可逆状态**：`FullyFilled`、`Rejected` 是终态，不可再转换
- **Canceled** 后不应再有该订单的成交回报（但在并发场景下需防范）
- 每次状态变更都应记录日志，便于审计和调试

### 1.3 并发场景下的一致性

#### 问题描述

在多线程或异步处理环境下，多个事件可能几乎同时发生，导致竞态条件。

#### 竞态场景 1：两笔对敲订单几乎同时到达

```
T1 (Thread A): 收到买单 ORD_001 (SH001, 600030, B, 500股, 25.50)
T1 (Thread B): 收到卖单 ORD_002 (SH001, 600030, S, 300股, 25.40)
```

如果两个线程同时进行对敲检测：
- Thread A：查找 SH001+600030 是否有反向订单 → 无 → 准备转发
- Thread B：查找 SH001+600030 是否有反向订单 → 无 → 准备转发

结果：**两笔都被转发到交易所**，对敲风控完全失效！

**解决方案**：
- 方案 1：**按 (shareHolderId, market, securityId) 组合加锁**，保证同一股东号 + 同一股票的订单串行处理
- 方案 2：**单线程事件循环**（如 Actor 模型），同一标的的订单由同一处理单元处理
- 方案 3：**无锁队列 + 分区处理**，按 hash(shareHolderId + securityId) 分配到固定处理线程

**推荐方案 3**：既保证正确性，又兼顾性能（不同标的可并行处理）。

#### 竞态场景 2：内部撮合与交易所撮合的竞争

```
T1: 买单 ORD_001 已转发到交易所
T2: 卖单 ORD_002 到达，检测到对敲
    → 系统发起内部撮合
    → 同时向交易所发送撤单请求（撤回 ORD_001）
T3: 交易所先撮合了 ORD_001（与其他卖家成交），然后收到我们的撤单请求
    → 撤单失败（或部分撤单）
```

此时系统内部已经生成了成交回报，但实际上 ORD_001 在交易所也产生了成交，可能导致**重复成交**。

**解决方案**：
- 内部撮合应标记为**暂定（Tentative）**状态
- 等到交易所撤单回报到达后，再最终确认内部撮合结果：
  - 撤单成功：内部撮合全额有效
  - 撤单部分成功（已成交 N 股）：内部撮合只对 `撤回数量` 部分有效
  - 撤单失败（全部已成交）：内部撮合无效，需回滚

> **简化方案**（基础目标阶段）：假设交易所是我们自己模拟的，可以保证撤单一定在撮合之前处理，避免竞态。在高级目标中再处理这种复杂情况。

#### 竞态场景 3：撤单与成交同时发生

```
T1: 客户发送撤单请求，撤回 ORD_001
T2: 同时交易所发来 ORD_001 的成交回报
```

系统应如何处理？
- 如果先收到成交回报：标记 ORD_001 已（部分/全部）成交，后续撤单只能撤回未成交部分
- 如果先收到撤单确认：标记 ORD_001 已撤销，后续成交回报应被忽略或告警

**关键原则**：以交易所的最终状态为准，本系统被动接受交易所的回报。

## 2. 撮合算法难题

### 2.1 成交价生成算法

内部撮合时，两笔对敲订单的价格可能不同，成交价如何确定？

#### 场景

```
买方订单: 25.50 元 × 500 股
卖方订单: 25.40 元 × 300 股
买价 25.50 ≥ 卖价 25.40，可成交
→ 成交价应该是多少？
```

#### 方案分析

| 方案 | 定价规则 | 成交价 | 优点 | 缺点 |
|------|---------|--------|------|------|
| A. 被动方价格 | 取先到订单的价格 | 若买先到：25.50 | 符合交易所惯例 | 需准确记录时间顺序 |
| B. 主动方价格 | 取后到订单的价格 | 若卖后到：25.40 | 后到的"吃单方"获得其报价 | 不常见 |
| C. 中间价 | (买价 + 卖价) / 2 | 25.45 | 对双方公平 | 可能产生非法价格精度 |
| D. 行情约束价 | 参考 bidPrice/askPrice | 取决于行情 | 最贴近市场 | 依赖行情数据 |

**推荐策略**：

- **基础目标（无行情数据）**：采用**方案 A（被动方价格 / 先到订单价格）**
  - 理由：与交易所的连续竞价规则一致，先到的挂单作为被动方，后到的吃单以被动方价格成交
  - 也就是说，先到的订单在"等待"，后到的订单在"吃"它，成交价以先到方的价格为准

- **高级目标（有行情数据）**：采用**方案 D（行情约束价）**
  - 需要确保成交价落在 `[bidPrice, askPrice]` 之间
  - 如果先到方价格在合理范围内，优先使用先到方价格
  - 如果超出范围，调整到边界值

#### 中间价的精度问题

若采用方案 C：
```
买价 25.51，卖价 25.40
中间价 = (25.51 + 25.40) / 2 = 25.455
```
A 股最小变动价位为 0.01 元，25.455 是非法价格。需要进行取整：
- 向买方有利（向下取整）：25.45
- 向卖方有利（向上取整）：25.46
- 四舍五入：25.46

这增加了复杂性，因此不推荐方案 C 作为基础实现。

### 2.2 零股处理

#### 零股成交的来源

在内部撮合中，零股可能由部分成交产生：

```
买方: 500 股 @ 25.50
卖方: 350 股 @ 25.40

成交: 350 股（整手 300 + 零股 50）
买方剩余: 150 股
```

成交的 350 股中包含零股部分（50 股）——这是合法的，因为成交可以产生任意数量。

#### 零股场景枚举

| 场景 | 买方数量 | 卖方数量 | 成交数量 | 是否含零股 | 处理方式 |
|------|---------|---------|---------|----------|---------|
| 整手 vs 整手 | 500 | 300 | 300 | 否 | 正常处理 |
| 整手 vs 零股 | 500 | 350 | 350 | 是 | 允许，成交中含零股 |
| 零股 vs 整手 | 150 | 300 | 150 | 是 | 允许 |
| 零股 vs 零股 | 150 | 80 | 80 | 是 | 允许 |

#### 零股剩余的处理

当撮合后产生零股剩余时：

```
买方: 500 股 @ 25.50
卖方: 350 股 @ 25.40
成交: 350 股

买方剩余: 150 股（含 50 股零股）
```

买方剩余 150 股如何处理？
- **转发到交易所**：150 股不是 100 的整数倍，若作为新买单提交，交易所可能拒绝（买入必须整手）
- **内部保留**：150 股继续留在内部订单簿等待下一次对敲配对
- **拆分处理**：整手部分（100 股）可转发，零股部分（50 股）保留或以其他方式处理

**推荐方案**：内部撮合后，剩余部分作为一个整体继续留在系统内部的订单簿中，等待后续配对或客户撤单。不拆分转发。

### 2.3 部分成交的连续配对

#### 问题描述

一笔大单可能需要与多笔小单依次配对：

```
已有挂单（SH001, 600030）:
  卖1: 200 股 @ 25.40
  卖2: 300 股 @ 25.45

新到买单: SH001, 600030, B, 600 股, 25.50

对敲检测: 买单与 卖1、卖2 都构成对敲
```

**撮合过程**（价格优先）：
1. 先与卖1 配对（价格更低，优先）：成交 200 股 @ 25.40
2. 买方剩余 400 股，再与卖2 配对：成交 300 股 @ 25.45
3. 买方最终剩余 100 股，无更多配对，保留在系统中

**关键设计点**：
- 每次配对生成独立的成交回报（带唯一 `execId`）
- 同一笔买单可以有多笔成交回报
- 需要维护订单的**累计成交数量**（`cumQty`），判断是否全部成交

### 2.4 成交回报的生成

每次撮合需要生成**两份**成交回报（买方一份、卖方一份），且共享同一个 `execId`：

```json
// 买方成交回报
{
  "clOrderId": "ORD_001",
  "market": "XSHG",
  "securityId": "600030",
  "side": "B",
  "qty": 500,
  "price": 25.50,
  "shareHolderId": "SH001",
  "execId": "EXEC_000001",
  "execQty": 300,
  "execPrice": 25.40
}

// 卖方成交回报
{
  "clOrderId": "ORD_002",
  "market": "XSHG",
  "securityId": "600030",
  "side": "S",
  "qty": 300,
  "price": 25.40,
  "shareHolderId": "SH001",
  "execId": "EXEC_000001",
  "execQty": 300,
  "execPrice": 25.40
}
```

注意：
- `execQty` 和 `execPrice` 相同（同一笔撮合）
- `clOrderId`、`qty`、`price` 各自反映原始订单信息
- `execId` 相同，标识这是同一笔撮合的两个方向

## 3. 系统设计难题

### 3.1 模块间耦合与消息传递

#### 问题描述

系统由多个模块组成（Gateway、Validator、RiskControl、MatchingEngine、ExchangeProxy 等），模块之间需要传递消息。如何设计模块间的通信方式？

#### 方案对比

| 方案 | 描述 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| 直接函数调用 | 模块之间通过接口直接调用 | 简单直接、低延迟 | 耦合度高、难以异步化 | 小型系统、原型验证 |
| 事件/回调 | 模块注册回调函数，事件触发时调用 | 解耦、灵活 | 回调地狱、调试困难 | 中等复杂度系统 |
| 消息队列 | 模块通过队列发送/接收消息 | 完全解耦、支持异步 | 增加延迟、内存开销 | 高并发、微服务 |
| Pipeline 管道 | 顺序通过一系列处理阶段 | 逻辑清晰、易扩展 | 分支逻辑复杂时不灵活 | 流水线处理 |

**推荐方案**：**Pipeline + 事件回调混合**

```
订单到达
  → [Gateway] 解析
    → [Validator] 校验
      → [RiskControl] 对敲检测
        ├─ 非对敲 → [ExchangeProxy] 转发 → 交易所
        └─ 对敲   → [MatchingEngine] 内部撮合
                     ├─ → 生成成交回报 → [Gateway] 输出
                     └─ → [ExchangeProxy] 发送撤单 → 交易所

交易所回报到达
  → [Gateway] 解析
    → [ReportHandler] 处理
      → [Gateway] 输出给客户
```

正向（订单处理）使用 Pipeline 模式（清晰的阶段流转），反向（回报处理）和分支逻辑（对敲后的撮合 + 撤单）使用事件回调。

### 3.2 可扩展性设计

任务书要求系统"可扩展"。以下是可能的扩展方向和对应的设计策略：

#### 扩展方向 1：新增校验规则

当前只有基本字段校验和对敲检测。未来可能新增：
- 涨跌停价格校验
- 单笔数量上限校验
- 持仓校验（不能卖出超过持有量）
- 频率限制（同一股东号短时间内过多下单）

**设计策略**：使用**责任链模式（Chain of Responsibility）**

```
class IOrderValidator {
    virtual ValidationResult validate(const Order& order) = 0;
};

class FieldValidator : public IOrderValidator { ... };     // 字段校验
class WashTradeDetector : public IOrderValidator { ... };  // 对敲检测
class PriceLimitValidator : public IOrderValidator { ... }; // 涨跌停校验（可选）

// 责任链
std::vector<std::unique_ptr<IOrderValidator>> validators;
for (auto& v : validators) {
    auto result = v->validate(order);
    if (!result.passed) return result;
}
```

#### 扩展方向 2：新增市场/品种

当前支持 XSHG/XSHE/BJSE 三个市场。未来可能新增：
- 港股通（XHKG）
- 期权市场（不同的交易规则）
- 可转债（不同的价格精度和交易单位）

**设计策略**：**市场规则抽象为配置**

```
struct MarketRules {
    std::string marketCode;
    double priceTick;          // 最小价格变动 0.01
    int lotSize;               // 手的大小 100
    double priceLimit;         // 涨跌幅 0.10
    bool allowOddLotBuy;       // 是否允许零股买入
};
```

#### 扩展方向 3：新增撮合策略

当前采用"价格优先、时间优先"。未来可能支持：
- 数量优先（大单优先成交）
- 最小剩余量优先

**设计策略**：**策略模式（Strategy Pattern）**

```
class IMatchingStrategy {
    virtual MatchResult match(Order& incoming, OrderBook& book) = 0;
};

class PriceTimePriority : public IMatchingStrategy { ... };
class PriceQuantityPriority : public IMatchingStrategy { ... };
```

### 3.3 撤单处理的复杂性（高级目标）

#### 问题分类

撤单请求可能面临的各种情况：

| 情况 | 原始订单状态 | 处理方式 |
|------|------------|---------|
| 订单尚在本系统内（未转发） | 内部等待 | 直接从内部订单簿移除 |
| 订单已发送到交易所，尚未确认 | 已发送 | 转发撤单到交易所 |
| 订单已被交易所确认 | 已确认 | 转发撤单到交易所 |
| 订单已部分成交 | 部分成交 | 撤回未成交部分 |
| 订单已全部成交 | 全部成交 | 拒绝撤单 |
| 订单已被拒绝 | 已拒绝 | 拒绝撤单 |
| 订单已被撤销 | 已撤销 | 拒绝撤单（重复撤单） |
| 订单正在内部撮合中 | 撮合进行中 | ⚠️ 复杂场景 |
| 订单不存在（编号无效） | N/A | 拒绝撤单 |

#### 撤单与内部撮合的交互

最复杂的情况：订单 A 正在与订单 B 进行内部撮合的过程中，订单 A 的撤单请求到达。

```
T1: 买单 A 到达，通过风控，转发到交易所
T2: 卖单 B 到达，对敲检测命中，开始内部撮合
    → 状态变更为 "InternalMatching"
    → 生成成交回报...
T3: 撤单请求到达，要求撤回订单 A
    → 此时 A 正在内部撮合中！
```

**解决方案**：
- 在内部撮合期间，订单状态标记为"锁定"状态
- 撤单请求入队等待，直到撮合完成后再处理
- 如果撮合后 A 还有剩余未成交部分，可以对剩余部分执行撤单

### 3.4 行情一致性约束（高级目标）

#### 问题描述

高级目标要求"撮合时需要保证买价、卖价和对手方价格保持一致"。即：

1. 内部撮合的成交价不能偏离当前市场行情太远
2. 行情数据有延迟，系统使用的行情可能不是最新的

#### 行情延迟的影响

```
T1: 行情快照 → bidPrice=25.45, askPrice=25.50
T2: 市场实际行情变化 → bidPrice=25.60, askPrice=25.65
T3: 系统进行内部撮合，使用 T1 的行情 → 成交价 25.47
T4: 新行情快照到达 → bidPrice=25.60, askPrice=25.65
```

此时 T3 时刻的成交价 25.47 已低于当前市场买价 25.60，是否合理？

**设计策略**：
- **最宽容**：只要在撮合**瞬间**的行情范围内即可（使用当时最新的行情快照）
- **较严格**：成交后校验是否仍在行情范围内，若偏离过大则告警
- **最严格**：使用行情的买卖价差中间值作为唯一可接受成交价

**推荐**：采用最宽容方案——使用撮合时刻可用的最新行情快照作为约束条件。行情数据本身就有延迟，在合理范围内接受延迟是行业惯例。

#### 行情缺失的处理

如果某只股票当前没有行情数据（如新上市、停牌后复牌等），内部撮合如何定价？

- 方案 1：拒绝内部撮合，将两笔订单都发送到交易所（放弃对敲风控）
- 方案 2：回退到无行情模式的定价策略（先到方价格）

**推荐方案 2**：有行情用行情约束，无行情则回退到基本策略。

### 3.5 ExchangeProxy 的模拟策略

本项目中"交易所"实际上需要自行模拟。ExchangeProxy 模块需要决定如何模拟交易所行为。

#### 模拟策略选项

| 策略 | 描述 | 复杂度 | 建议 |
|------|------|--------|------|
| 简单确认 | 收到订单即回复确认，撤单即确认 | ★☆☆ | 基础目标足够 |
| 简单撮合 | 维护一个简单的订单簿，实现基本撮合 | ★★☆ | 中等目标 |
| 完整撮合 | 完整的交易所撮合引擎 | ★★★ | 可选 |
| 外部模拟器 | 通过网络协议与外部模拟交易所通信 | ★★★ | 演示效果好 |

**推荐**：基础目标使用"简单确认"，高级目标时视需要提升。

## 4. 性能难题

### 4.1 延迟分析

#### 关键路径

```
交易系统延迟 = 网络接收 + JSON解析 + 校验 + 风控检测 + 决策 + JSON序列化 + 网络发送
```

各环节的典型耗时（参考值）：

| 环节 | 典型耗时 | 优化空间 |
|------|---------|---------|
| 网络 I/O | 1-100 μs | epoll/IOCP、零拷贝 |
| JSON 解析 | 5-50 μs | 使用 rapidjson / simdjson |
| 字段校验 | < 1 μs | 查表法 |
| 对敲检测（哈希查找） | < 1 μs | 预计算 hash key |
| 撮合逻辑 | 1-10 μs | 优化数据结构 |
| JSON 序列化 | 5-50 μs | 预分配缓冲区 |
| 内存分配 | 0.5-5 μs | 对象池 |

**瓶颈**：JSON 解析/序列化是最大的延迟来源。在极致优化场景下，可考虑使用二进制协议替代 JSON。

#### 低延迟优化方向

1. **减少内存分配**
   - 使用对象池预分配 Order/Report 对象
   - 使用 `std::string_view` 替代 `std::string` 避免拷贝
   - JSON 解析使用 SAX 模式（流式解析）替代 DOM 模式

2. **减少锁竞争**
   - 按标的分区（hash 分片），不同标的互不干扰
   - 使用无锁数据结构（lock-free queue）传递消息
   - 避免全局锁

3. **减少系统调用**
   - 批量处理 I/O（readv/writev）
   - 避免频繁的系统调用（如 `std::chrono::now()`）

4. **CPU 亲和性**
   - 关键路径线程绑定到特定 CPU 核心
   - 避免上下文切换

### 4.2 吞吐量分析

#### 目标

高峰期 A 股市场的总订单量可达十万/秒级别。本项目作为模拟系统，合理的目标是：
- 基础目标：≥ 1,000 笔/秒
- 高级目标：≥ 100,000 笔/秒

#### 吞吐量瓶颈

| 瓶颈点 | 原因 | 解决方案 |
|--------|------|---------|
| JSON 解析 | CPU 密集 | 使用 SIMD 加速的 simdjson |
| 订单簿查找 | 不当数据结构 | 使用 hashmap 代替线性查找 |
| 锁竞争 | 全局锁序列化处理 | 分区处理 |
| 内存分配 | 频繁 new/delete | 对象池 + 内存预分配 |
| I/O | 同步阻塞 | 异步 I/O / epoll |

#### 数据结构对吞吐量的影响

对敲检测中最频繁的操作是：**按 (shareHolderId, market, securityId) 查找是否存在反向挂单**

| 数据结构 | 查找复杂度 | 内存布局 | 适用场景 |
|---------|----------|---------|---------|
| `std::unordered_map` | O(1) 平均 | 链式，缓存不友好 | 通用 |
| `robin_map` (开放寻址) | O(1) 平均 | 紧凑，缓存友好 | 高性能 |
| `std::map` (红黑树) | O(log n) | 指针跳转，缓存不友好 | 需要有序遍历 |
| 自定义数组哈希 | O(1) 平均 | 最紧凑 | 极致性能 |

**推荐**：
- 基础目标：`std::unordered_map`，用组合 key `shareHolderId + "|" + market + "|" + securityId` 定位
- 高级目标：考虑 `robin_map` 或自定义哈希表

### 4.3 内存管理

#### 订单簿的内存开销估算

每笔 Order 对象的大致大小：

```
clOrderId:     16 字节
market:         4 字节
securityId:     6 字节
side:           1 字节
qty:            4 字节
price:          8 字节
shareHolderId: 10 字节
状态/时间戳等: 16 字节
填充对齐:       ~3 字节
──────────────────────────
合计约 ~68 字节（对齐后约 72-80 字节）
```

若系统中同时有 100,000 笔未成交订单：
- 内存占用：100,000 × 80 = 8 MB
- 加上哈希表开销（约 2 倍）：~16 MB
- 完全在合理范围内

**注意**：如果存储历史记录（高级目标），日积月累的数据量才是内存/存储的主要关注点。

### 5.2 并发测试

并发场景的测试需要特殊策略：

#### 方法 1：确定性并发测试

- 使用固定顺序的输入序列模拟并发效果
- 可预知预期结果，易于验证
- 适合单元测试

```
// 模拟两笔同时到达的订单
orderQueue.push(buyOrder);   // T=0
orderQueue.push(sellOrder);  // T=0 (同一时间戳)
processor.processAll();
// 验证结果
```

#### 方法 2：压力测试

- 多线程同时提交大量订单
- 检查不变量（invariants）：
  - 不存在同一股东号的买卖同时在交易所成交
  - 所有成交回报都能对应到原始订单
  - 订单数量守恒（原始数量 = 成交数量 + 剩余数量 + 撤销数量）
  - 不存在重复的 execId

#### 方法 3：模糊测试（Fuzz Testing）

- 随机生成各种格式的JSON输入
- 确保系统不会崩溃（robustness）
- 验证错误处理的完整性

### 5.3 性能基准测试方案

#### 延迟测量

// 在关键路径上插入高精度计时

- 平均延迟
