## 二、系统角色与核心概念

### 2.1 我们的系统是什么？

本系统模拟的是一个 **券商委前风控网关**（不是交易所）。在真实证券交易中：

```
投资者 → [我们的系统] → 交易所
               ↑
         拦截对敲、内部撮合
```

核心定位：

- 正常订单：**透明转发**给交易所，不做任何干预
- 对敲订单：**拦截 + 内部撮合**，避免违规委托到达交易所
- 同时提供**实时监控**功能，让企业端观察系统运行状态

### 2.2 什么是"对敲"（Wash Trade）？

同一股东账号（`shareHolderId`），在同一市场（`market`），对同一证券（`securityId`），同时挂出买单和卖单，自己和自己成交——这就是对敲。

判定的复合键：`shareHolderId | market | securityId`（代码中称为 `washTradeKey`）

### 2.3 对敲检测后为什么要"内部撮合"？

如果仅仅拒绝对敲订单，会破坏客户的合法交易需求（例如多策略基金的不同策略独立下单）。正确做法是：

1. 拦截这对订单，不让其到达交易所
2. 在系统内部完成撮合（按先到方的价格成交）
3. 如果早先到的那笔已经发到了交易所，还要发撤单

## 三、核心处理流程（Pipeline）

这是整个系统最核心的流程，也是你最需要理解的部分。入口是 `OrderProcessingService.processOrder()`。

```
客户提交委托 (OrderDTO)
        │
        ▼
  ┌─────────────────────┐
  │  1. 字段校验          │  OrderValidationChain
  │  (6条规则, 责任链)     │  → 必填字段/市场/证券/方向/数量/价格
  └──────────┬──────────┘
             │ 校验失败 → 返回 REJECTED (RejectReportDTO)
             │ 校验通过 ↓
  ┌─────────────────────┐
  │  2. 构建内部订单       │  InternalOrder.fromDTO()
  │  + 生成 washTradeKey  │
  └──────────┬──────────┘
             │
  ┌─────────────────────┐
  │  3. 对敲检测          │  WashTradeDetector.detect()
  │  (在 washTradeKey    │  → 查找同key下反向且价格可成交的订单
  │   粒度的锁内执行)     │
  └──────────┬──────────┘
             │
     ┌───────┴────────┐
     │                │
  未检出             检出对敲
     │                │
     ▼                ▼
  ┌────────┐    ┌──────────────────┐
  │ 4a.    │    │ 4b. 对敲拦截      │
  │ 注册索引│    │ + 内部撮合        │  MatchingEngine.match()
  │ 转发交易│    │ + 撤回已发送订单   │  ExchangeProxy.sendCancelRequest()
  │ 所     │    └──────────────────┘
  └───┬────┘              │
      │                   │
      ▼                   ▼
  FORWARDED         WASH_TRADE_HANDLED
  (ConfirmReport)   (RejectReport + ExecReports + CancelConfirms)
      │                   │
      └─────────┬─────────┘
                ▼
  ┌─────────────────────┐
  │  5. 回报分发          │  ReportDispatcher → WebSocketPublisher
  │  (通过 WebSocket      │  → /topic/reports/confirm
  │   推送给前端)          │  → /topic/reports/reject
  │                      │  → /topic/reports/execution
  │                      │  → /topic/alerts/washtrade
  └──────────────────────┘
```

### 3.1 校验模块详解

采用 **责任链模式**：每条规则实现 `ValidationRule` 接口，返回 `Optional<RejectReportDTO>`（空 = 通过，非空 = 拒绝）。

```java
// ValidationRule 接口
Optional<RejectReportDTO> validate(OrderDTO order);
```

`OrderValidationChain` 通过 Spring 注入所有规则（按 `@Order` 排序），短路执行——第一个失败就立即返回。

| 顺序 | 规则类              | 校验内容                            |
| ---- | ------------------- | ----------------------------------- |
| 1    | `RequiredFieldRule` | `clOrderId` 和 `shareHolderId` 非空 |
| 2    | `MarketRule`        | market 必须是 XSHG/XSHE/BJSE        |
| 3    | `SecurityIdRule`    | securityId 为 6 位数字              |
| 4    | `SideRule`          | side 必须是 B 或 S                  |
| 5    | `QuantityRule`      | qty > 0 且为 100 的整数倍           |
| 6    | `PriceRule`         | price > 0                           |

**扩展方式**：新建一个实现 `ValidationRule` 的类，加上 `@Component` 和 `@Order(n)`，Spring 会自动注入到责任链中。

### 3.2 对敲检测模块详解

核心是 `OrderIndex` —— 一个精心设计的嵌套索引结构：

```
ConcurrentHashMap<washTradeKey, EnumMap<OrderSide, TreeMap<价格, Deque<InternalOrder>>>>
```

用自然语言描述就是：按「股东+市场+证券」分组 → 按「买/卖」分组 → 按「价格」排序 → 每个价位一个队列。

检测逻辑（`findCounterOrders`）：

- 对买单，找所有 **卖出价 ≤ 买入价** 的卖单（使用 TreeMap 的 `headMap`）
- 对卖单，找所有 **买入价 ≥ 卖出价** 的买单（使用 TreeMap 的 `tailMap`）
- 找到就说明存在对敲

**并发安全**：每个 `washTradeKey` 有独立的锁（`getLock()`），不同股东/证券的订单可以完全并行处理。

### 3.3 撮合引擎详解

`MatchingEngine.match()` 对对敲命中的订单做内部配对：

- 遍历反向订单列表（counterOrders）
- 每笔成交量 = min(发起方剩余, 对手方剩余)
- 成交价 = **被动方（先到方）的价格**
- 每次撮合生成两条 `ExecReportDTO`（买方一条、卖方一条），共享同一个 `execId`
- 更新双方的 `remainQty` 和 `cumQty`
- 如果对手方完全成交，从 `OrderIndex` 中移除

### 3.4 交易所模拟器

当前为 **Level 1（简单确认模式）**：

- `receiveOrder()`：收到订单就立即返回确认回报
- `receiveCancelRequest()`：收到撤单就立即返回撤单确认

与真实交易所的差异：真实交易所会有撮合过程，可能部分成交、可能拒绝。后续可升级为 Level 2（含简单撮合逻辑）。

### 3.5 回报分发

`ReportDispatcher` 根据处理结果类型，调用 `WebSocketPublisher` 推送到不同的 STOMP 主题：

| 主题                       | 内容           |
| -------------------------- | -------------- |
| `/topic/reports/confirm`   | 交易所确认回报 |
| `/topic/reports/reject`    | 拒绝回报       |
| `/topic/reports/execution` | 成交回报       |
| `/topic/alerts/washtrade`  | 对敲警报       |
| `/topic/reports/cancel`    | 撤单确认       |

### 4.4 Vite 代理配置

`vite.config.ts` 配置了开发代理：

```typescript
proxy: {
  '/api': { target: 'http://localhost:8080', changeOrigin: true },
  '/ws':  { target: 'http://localhost:8080', ws: true },
}
```

前端 `localhost:5173` 上所有 `/api` 和 `/ws` 请求会透明转发到后端 `localhost:8080`，无需处理跨域。

| **用户认证**             | 登录/角色区分                        | 低     | 设计上预留了 trader/admin 区分                        |
| **线程安全完善**         | orderRegistry 线程安全               | 中     | 当前使用普通 LinkedHashMap                            |

### 6.3 代码质量注意事项

1. **OrderProcessingService.orderRegistry**：使用 `LinkedHashMap`（非线程安全），目前在并发场景下存在风险。后续应改为 `ConcurrentLinkedHashMap` 或加锁。
2. **ExchangeSimulator**：Level 1 模式下所有订单都是立即确认，不模拟真实的排队和撮合过程，仅用于验证基础流程。
3. **数据未持久化**：当前所有数据存储在内存中，重启后丢失。Repository 接口已定义但未被 Service 层调用。

## 九、开发指引

### 9.1 如何新增一条校验规则

1. 在 `validator/rules/` 下新建类，例如 `AccountRule.java`
2. 实现 `ValidationRule` 接口
3. 加 `@Component` 和 `@Order(7)`（顺序取决于你希望它何时执行）
4. 实现 `validate(OrderDTO)` 方法
5. 不需要修改其他任何代码——Spring 会自动将其注入到责任链中

### 9.2 如何新增 REST 接口

1. 在 `gateway/controller/` 下新建或修改 Controller 类
2. 使用 `@RestController` + `@RequestMapping`
3. 添加 Swagger 注解 `@Tag` / `@Operation`
4. 注入所需的 Service

### 9.3 如何新增前端页面

1. 在 `frontend/src/views/` 下新建 `.vue` 文件
2. 在 `router/index.ts` 的 `children` 数组中新增路由
3. 在 `layout/MainLayout.vue` 的 `menuItems` 数组中新增菜单项

### 9.4 如何接入数据库持久化

1. 在 `OrderProcessingService` 中注入 `OrderRepository` 和 `ExecutionRepository`
2. 在订单处理流程的适当位置调用 `repository.save(entity)`
3. 将 `InternalOrder` 转为 `OrderEntity`，将 `ExecReportDTO` 转为 `ExecutionEntity`
4. 建议使用 `@Async` 异步写入，避免阻塞主流程

## 十、总结

当前项目框架已经建立了一个 **端到端可运行** 的基础系统：

- 后端 42 个 Java 源文件，覆盖了完整的订单处理管线（校验 → 对敲检测 → 撮合/转发 → 回报分发）
