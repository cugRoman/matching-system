# 股票交易对敲撮合系统核心算法（修订版）

## 1. 算法目标

实现符合 A 股市场“价格优先、时间优先”原则的竞价撮合，并严格遵守对敲风控规则：

- **对敲定义**：同一股东号在同一支股票上同时存在买入和卖出委托。
- **风控处理**：在撮合前，一次性检测出所有可能构成对敲的订单，将其整体移入失败列表（生成非法回报），剩余合法订单再进行撮合。
- **撮合规则**：买入订单按价格降序、同价按时间升序排列；卖出订单按价格升序、同价按时间升序排列。当最优买单价格 ≥ 最优卖单价格时，即可成交。

## 2. 数据结构定义

### 2.1 订单核心字段（按报文顺序）

| 字段名        | 类型      | 说明                               |
| ------------- | --------- | ---------------------------------- |
| clOrderId     | String    | 订单唯一编号                       |
| shareHolderId | String    | 股东号（虚拟账户）                 |
| securityId    | String    | 股票代码                           |
| side          | char      | 'B' 买入 / 'S' 卖出                |
| price         | double    | 委托价格                           |
| qty           | int       | 委托数量（整数）                   |
| timestamp     | long      | 订单到达时间（毫秒，用于时间优先） |
| **status**    | int       | **状态标识**（见下文）             |

**状态标识（status）定义**：

| 值 | 含义                     | 适用列表                         |
|----|--------------------------|----------------------------------|
| 0  | 未交易（未挂载）         | 请求列表                         |
| 1  | 未交易并挂载             | 交易所挂单列表                   |
| 2  | 部分交易并挂载           | 交易所挂单列表                   |
| 3  | 已完全成交               | 撮合成功列表（历史）             |
| 4  | 非法（对敲）             | 撮合失败列表                     |

**说明**：  
- 请求列表中的订单 status = 0（刚提交，尚未处理）。  
- 交易所挂单列表中的订单 status = 1 或 2，表示已挂载至交易所。  
- 撮合成功后，订单从交易所列表中移除，对应的成交记录进入撮合成功列表（status 不再保留）。  
- 非法对敲的订单直接进入撮合失败列表，status = 4。

### 2.2 系统维护的全局列表（按股票代码分别管理）

| 列表名称               | 存储内容                                                                 | 排序规则                     |
|------------------------|--------------------------------------------------------------------------|------------------------------|
| `buyRequests`          | 新提交但尚未处理的买入订单（status = 0）                                 | 价格降序，同价时间升序       |
| `sellRequests`         | 新提交但尚未处理的卖出订单（status = 0）                                 | 价格升序，同价时间升序       |
| `exchangeBuys`         | 已挂载到交易所的买入订单（status = 1 或 2）                              | 价格降序，同价时间升序       |
| `exchangeSells`        | 已挂载到交易所的卖出订单（status = 1 或 2）                              | 价格升序，同价时间升序       |
| `tradeSuccesses`       | 历史所有成交记录（每次成交生成一条记录，含买卖双方信息）                 | 按成交时间倒序               |
| `tradeIllegals`        | 因对敲等原因被拒绝的订单（每条记录对应一笔委托，status = 4）             | 按拒绝时间倒序               |

### 2.3 各列表记录字段

#### 2.3.1 请求列表（`buyRequests` / `sellRequests`）和交易所挂单列表（`exchangeBuys` / `exchangeSells`）

每条记录包含订单核心字段（clOrderId, shareHolderId, securityId, side, price, qty, timestamp, status）。

#### 2.3.2 撮合成功列表（`tradeSuccesses`）

| 字段名           | 类型   | 说明                                   |
| ---------------- | ------ | -------------------------------------- |
| execId           | String | 成交唯一编号（自动生成）               |
| securityId       | String | 股票代码                               |
| buyOrderId       | String | 买方订单号（clOrderId）                |
| sellOrderId      | String | 卖方订单号（clOrderId）                |
| buyShareHolderId | String | 买方股东号                             |
| sellShareHolderId| String | 卖方股东号                             |
| execQty          | int    | 成交数量                               |
| execPrice        | double | 成交价格                               |
| execTime         | long   | 成交时间（毫秒）                       |

#### 2.3.3 撮合失败列表（`tradeIllegals`）

| 字段名        | 类型   | 说明                                 |
| ------------- | ------ | ------------------------------------ |
| clOrderId     | String | 订单唯一编号                         |
| shareHolderId | String | 股东号                               |
| securityId    | String | 股票代码                             |
| side          | char   | 买卖方向                             |
| price         | double | 委托价格                             |
| qty           | int    | 委托数量                             |
| rejectCode    | int    | 拒绝代码（如 1001 表示对敲）         |
| rejectTime    | long   | 拒绝时间（毫秒）                     |
| status        | int    | 固定为 4（非法）                     |

## 3. 算法流程（每支股票独立执行）

### 3.1 输入
- 当前请求列表（`buyRequests` + `sellRequests`，status = 0）
- 当前交易所挂单（`exchangeBuys` + `exchangeSells`，status = 1 或 2）

### 3.2 输出
- 更新后的交易所挂单（`exchangeBuys` / `exchangeSells`）
- 追加的撮合成功记录（`tradeSuccesses`）
- 追加的撮合失败记录（`tradeIllegals`）

### 3.3 步骤详解

#### 步骤1：合并待处理订单
将该股票的所有订单（请求列表 + 交易所挂单）合并为一个临时列表 `allOrders`。注意保留每个订单的原始 status，但撮合逻辑中 status 仅用于区分来源，检测非法时需考虑所有订单。

#### 步骤2：按股东号分组检测非法对敲
1. 遍历 `allOrders`，按 `shareHolderId` 分组。
2. 对于每个股东号，检查该组中是否**同时包含**买入（`side = 'B'`）和卖出（`side = 'S'`）订单。
3. 如果同时包含：
   - 将该组**所有订单**标记为非法（status = 4）。
   - 将这些订单从 `allOrders` 中移除。
   - 为每个被移除的订单生成一条非法记录，追加到 `tradeIllegals` 中（包含订单信息、rejectCode = 1001、rejectTime 当前时间、status = 4）。
4. 如果组内只有单一方向的订单，则保留为合法订单（status 保持不变：原请求列表订单 status = 0，原交易所挂单 status = 1 或 2）。

#### 步骤3：构建合法撮合队列
将剩余合法订单分别放入两个优先队列：

- **买单队列** `buyQueue`：按**价格降序**，同价按**时间升序**。
- **卖单队列** `sellQueue`：按**价格升序**，同价按**时间升序**。

注意：队列中的订单仍携带原 status（0/1/2），用于后续判断是否来自交易所。

#### 步骤4：执行撮合匹配
```
while (buyQueue 不为空 && sellQueue 不为空) {
    buy = buyQueue.peek(); // 当前最优买单
    sell = sellQueue.peek(); // 当前最优卖单

    if (buy.price < sell.price) {
        break; // 无法继续匹配
    }

    // 弹出两个订单
    buy = buyQueue.poll();
    sell = sellQueue.poll();

    // 计算成交数量
    long tradeQty = Math.min(buy.qty, sell.qty);

    // 成交价生成规则（采用卖方价格）
    double execPrice = sell.price;

    // 生成成交记录，追加到 tradeSuccesses
    生成成交记录(buy, sell, tradeQty, execPrice);

    // 更新剩余数量
    buy.qty -= tradeQty;
    sell.qty -= tradeQty;

    if (buy.qty > 0) {
    // 部分成交的买单重新入队，并将其状态更新为 2（部分交易并挂载），因为后续它将成为交易所挂单
    buy.status = 2;
        buyQueue.add(buy);
    }
    if (sell.qty > 0) {
        sell.status = 2;
        sellQueue.add(sell);
    }
}
```
**说明**：
- 当订单来自请求列表（原 status = 0）且首次成交后剩余部分时，其状态变为 2（部分交易并挂载），表示后续将挂载到交易所。
- 当订单来自交易所挂单（原 status = 1 或 2）且剩余部分时，保持或更新为 status = 2。

#### 步骤5：更新交易所挂单并清空请求列表
- 将 `buyQueue` 和 `sellQueue` 中所有剩余订单（包括部分成交后剩余的）分别放入 `exchangeBuys` 和 `exchangeSells`。这些订单的 status 已在上一步设置为 1（若完全未成交且原 status=0 则设为 1）或 2（部分成交）。
- 清空 `buyRequests` 和 `sellRequests`（所有请求已处理完毕）。

## 4. 关键点说明

- **通过 status 和 side 区分列表位置**：
  - 请求列表中的订单 status = 0，交易所挂单中的订单 status = 1 或 2。
  - 撮合成功列表记录成交事件，不保留订单本身；撮合失败列表记录被拒订单，status = 4。
  - 撮合过程中，通过 side 区分买卖，放入不同队列。
- **对敲检测的完整性**：在撮合前一次性清除所有可能构成对敲的订单，确保撮合过程只处理合法订单。
- **撤单隐含机制**：使用交易所挂单参与撮合相当于从交易所撤单，匹配结束后未成交部分重新挂回。
- **时间优先实现**：优先队列中同价订单按时间戳升序排列，保证先到先得。
- **成交价生成**：采用卖方价格，可灵活调整。

## 5. 演示示例（股票“600519”）

### 初始状态（请求列表 + 交易所挂单）

| clOrderId | shareHolderId | side | price | qty | timestamp | status |
|-----------|---------------|------|-------|-----|-----------|--------|
| B001      | SH001         | B    | 1800  | 100 | t1        | 0      |
| B002      | SH002         | B    | 1795  | 200 | t2        | 1      |
| B003      | SH003         | B    | 1800  | 150 | t3        | 0      |
| S001      | SH004         | S    | 1790  | 80  | t4        | 0      |
| S002      | SH005         | S    | 1800  | 100 | t5        | 1      |
| S003      | SH006         | S    | 1800  | 120 | t6        | 1      |
| B004      | SH007         | B    | 1810  | 50  | t7        | 0      |
| S004      | SH007         | S    | 1805  | 50  | t8        | 0      |
| B005      | SH007         | B    | 1800  | 100 | t9        | 0      |

注：status=1 表示已在交易所挂单未成交，status=0 表示新请求。

### 步骤2 检测结果
- 股东号 SH007：同时存在买单 B004、B005 和卖单 S004 → 三笔订单全部移入失败列表，status=4。

### 步骤4 撮合过程
合法订单：B001(0)、B002(1)、B003(0)、S001(0)、S002(1)、S003(1)

1. B001(1800) vs S001(1790) → 成交 80 股 @1790，B001 剩余 20（状态变为 2），S001 完全成交（不再入队）。
2. B001(剩余20,2) vs S002(1800,1) → 成交 20 股 @1800，S002 剩余 80（状态保持或变为 2）。
3. B003(1800,0) vs S002(剩余80,2) → 成交 80 股 @1800，B003 剩余 70（状态 2）。
4. B003(剩余70,2) vs S003(1800,1) → 成交 70 股 @1800，S003 剩余 50（状态 2）。
5. B002(1795,1) 价格低于剩余卖单 S003(1800,2)，无法继续。

### 最终结果
- **交易所挂单**：
  - B002(1795,200, status=1)  —— 原交易所挂单未参与成交，保持 status=1
  - S003(剩余50,1800, status=2) —— 部分成交后剩余
- **撮合成功列表**：新增 4 条成交记录。
- **撮合失败列表**：新增 B004、B005、S004 三条非法记录，status=4。
